    读任何一行前，先知道数据和命令往哪走

    源码才是当前事实README 是重要参考，但可能滞后。当前 main.c 实际加载的是 AccomplishTest_GetMissionGraph()。
    100 Hz 是项目心跳SysTick 每 10 ms 产生一次节拍。传感、控制、任务和显示都围绕这个时间基准协作。
    分层不是文件夹装饰Application 决定“做什么”，Hardware 负责“怎么碰硬件”，System 提供时间与中断基础。

    命令与所有权链：谁有权让谁做事

    main初始化后永远循环；只负责把 App 与 Mission 接起来。
    App等待节拍、更新传感状态、通信、运动控制和调试输出。
    Mission根据按键/串口事件和动作完成状态，决定下一项任务。
    MotionManager统一选择直行、巡线、转向或制动，避免抢占电机。
    Hardware最终读 GPIO/编码器/MPU6050，写 PWM/OLED/串口。

    同时还有两条观察支线：State → Telemetry → car_debug.html 把运行数据送到电脑；State → DebugDisplay → OLED 把关键状态显示在车上。

    每个有效节拍的实际执行顺序：main → App_Update（Heading → Odometry → 通信 → MotionManager → 遥测/OLED）→ Mission_Update。Mission 在本拍末尾决定下一动作；新命令会在随后节拍由 MotionManager 持续执行。因此上图表达“谁向谁下命令”，不是函数在一拍内的先后顺序。

    推荐顺序：01 运行骨架 → 02 板级 I/O → 03 状态感知 → 04 轮速闭环 → 05 运动行为 → 06 通信 → 07 可观测性 → 08 状态机引擎 → 09 具体任务 → 10 工具链

    没有模块包含这个关键词。试试文件名、函数名、PID、状态机或串口。
