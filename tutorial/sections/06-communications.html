模块 06：串口命令与 K230 协议——从一个字节到整车动作

一句话职责：Serial 层可靠收发 UART 字节；BluetoothDebug 把 UART1 文本翻译成调试、任务和运动命令；K230Link 把 UART2 字节翻译成带校验的二进制帧。

输入：UART1 的无线 DAPLink 串口/网页字符流、UART2 的 K230 字节流、App 的 100 Hz 经过节拍、MotionManager 忙闲状态。BluetoothDebug 是早期接蓝牙模块留下的历史文件名，当前物理链路并不是蓝牙。

输出：电机/舵机/运动/遥测设置、给 Mission 的单槽信号、K230 READY/TARGET/CAPTURE/CAPTURE_ACK 数据。

依赖：TI UART DriverLib、SysConfig 生成宏、Motor、Servo、Heading、MotionManager、Telemetry。

调用者：UART 中断写入缓冲区；App_Init 初始化；App_Update 周期调用两个协议层并弹出信号和拍照结果。

总体位置：外部设备 → UART ISR → 环形缓冲 → Application 协议解析 → App/Mission/MotionManager/Telemetry。

1. 先看两条完全不同的通信链

电脑网页 → 无线 DAPLink 串口 → UART1 RX(PB7) → Serial1 环形缓冲 → BluetoothDebug 文本解析 → Motor / Servo / MotionManager / Telemetry / Mission
K230 → UART2 RX(PA22) → Serial2 环形缓冲 → K230Link 帧状态机 → TARGET 或 CAPTURE_ACK
网页 P5 → BluetoothDebug → K230Link_RequestCapture(5) → UART2 CAPTURE 帧 → K230 → CAPTURE_ACK → App → UART1 “OK CAP”

两条 UART 都是 115200 baud，但协议不能混用。UART1 面向人和网页，允许 L100、F500、C1 这种可读文本；UART2 面向另一颗处理器，使用 AA 55 帧头、长度和 CRC，能从连续二进制流中识别完整消息。

接线提醒：看到模块名 BluetoothDebug 不要去找蓝牙模块。当前 UART1 接无线 DAPLink 的串口桥，网页通过电脑浏览器的 Web Serial 使用它；UART2 才是 MCU 与 K230 之间的专用链路。

2. 初学者需要先知道

通用异步收发器用 TX、RX 两根信号线按约定波特率逐位发送字节。“异步”表示没有单独时钟线，双方必须预先约好 115200 等速率。

中断服务函数是硬件事件发生时由 CPU 临时跳去执行的函数。这里 UART 收到字节便进入 ISR，迅速把字节搬进内存，然后把复杂解析留给主循环。

易变对象告诉编译器：变量可能被当前代码看不见的执行路径修改。本项目的读写索引由 ISR 和主循环共享，所以索引声明为 volatile，避免编译器把旧值长期放在寄存器里。

环形缓冲区把固定数组首尾相接使用。逻辑索引一直递增，访问数组时用 index % size 绕回开头，因此中断可继续接收，主循环稍后再读。

单生产者单消费者表示只有 ISR 写入、只有主循环读出。职责单一使环形缓冲无需复杂锁；32 位 MCU 对对齐的 32 位索引读写也能一次完成。

可变参数函数可以接收数量不固定的参数。Serial1_Printf 的 ... 配合 va_list，把格式化结果先写进 192 字节数组，再发串口。

解析器逐字节记住“已经读到哪里”。文本解析器记命令字、符号和十进制数字；K230 状态机依次等待帧头、版本、类型、序号、长度、负载、CRC。

小端序规定多字节整数的低 8 位先传。K230 的 offsetX、offsetY 和拍照索引都用 low | high<<8 还原。

8 位循环冗余校验用多项式运算把一串字节压成 1 字节校验值。传输中任一位变化通常会导致校验不一致，整帧被丢弃。

void Serial1_Init(void)
{
    s_writeIndex = 0U;
    s_readIndex = 0U;
    Serial1_RxFlag = 0U;
    /*
     * PB7 使用上拉，避免蓝牙模块断开或未供电时 RX 悬空并产生伪数据。
     * 本配置必须在 SYSCFG_DL_init() 之后执行，以重新应用带上拉的外设输入功能。

    DL_GPIO_initPeripheralInputFunctionFeatures(
        GPIO_BLUETOOTH_UART_IOMUX_RX, GPIO_BLUETOOTH_UART_IOMUX_RX_FUNC,
        DL_GPIO_INVERSION_DISABLE, DL_GPIO_RESISTOR_PULL_UP,
        DL_GPIO_HYSTERESIS_DISABLE, DL_GPIO_WAKEUP_DISABLE);
    NVIC_ClearPendingIRQ(BLUETOOTH_UART_INST_INT_IRQN);
    NVIC_EnableIRQ(BLUETOOTH_UART_INST_INT_IRQN);
}

uint32_t Serial1_Available(void)
{
    return s_writeIndex - s_readIndex;
}

uint8_t Serial1_ReadByte(uint8_t *byte)
{
    if ((byte == NULL) || (s_readIndex == s_writeIndex)) return 0U;
    *byte = s_rxBuffer[s_readIndex % SERIAL1_RX_BUFFER_SIZE];
    s_readIndex++;
    if (s_readIndex == s_writeIndex) Serial1_RxFlag = 0U;
    return 1U;
}

void UART1_IRQHandler(void)
{
    while (!DL_UART_Main_isRXFIFOEmpty(BLUETOOTH_UART_INST))
    {
        uint8_t data = DL_UART_Main_receiveData(BLUETOOTH_UART_INST);
        s_rxBuffer[s_writeIndex % SERIAL1_RX_BUFFER_SIZE] = data;
        s_writeIndex++;
        if ((s_writeIndex - s_readIndex) > SERIAL1_RX_BUFFER_SIZE)
        {
            s_readIndex = s_writeIndex - SERIAL1_RX_BUFFER_SIZE;
        }
        Serial1_RxFlag = 1U;
    }
}

void UART2_IRQHandler(void)
{
    while (!DL_UART_Main_isRXFIFOEmpty(K230_INST))
    {
        uint8_t data = DL_UART_Main_receiveData(K230_INST);
        s_serial2RxBuffer[
            s_serial2WriteIndex % SERIAL2_RX_BUFFER_SIZE] = data;
        s_serial2WriteIndex++;
        if ((s_serial2WriteIndex - s_serial2ReadIndex) >
            SERIAL2_RX_BUFFER_SIZE)
        {
            s_serial2ReadIndex =
                s_serial2WriteIndex - SERIAL2_RX_BUFFER_SIZE;
        }
    }
}
  1. 第 125～129 行：UART1 ISR 用 while 一次排空硬件 RX FIFO；若一次中断已经积累多个字节，不只取第一个。
  2. 第 130～131 行：写索引对容量取模决定数组槽位，然后累计写索引加 1。
  3. 第 132～135 行：若写减读超过 1024，说明生产者追上消费者；代码把读索引推进到“最早仍保存的字节”，丢弃最旧数据，保留最新 1024 字节。
  4. 第 136～138 行：置接收提示并继续检查 FIFO，直到硬件为空才退出中断。
  5. 第 140～146 行：UART2 ISR 采用同一结构，只是实例、数组和容量不同。
  6. 第 147～153 行：UART2 超过 256 字节时也丢最旧数据。长 K230 流不会越界写数组，但若主循环长期不运行，协议帧开头可能被覆盖。
  7. 第 154～155 行：排空 FIFO 后退出中断。

难点拆解：为什么索引不主动归零

输入：writeIndex=1026，readIndex=1022，容量 1024。

过程：Available=4；真实数组位置分别通过 %1024 得到 2 和 1022。逻辑计数与物理槽位分开。

输出：无需判断“尾部是否绕到头部”，write-read 永远直接表示排队数量。

边界：uint32_t 最终会回绕，但无符号减法按模 2³² 运算；只要队列差始终被限制在容量内，回绕前后仍成立。

设计原因：单生产者 ISR 只写 write，单消费者主循环只写 read，共享修改最少，逻辑比“head/tail 留一个空槽”更直观。

static uint8_t K230Link_Crc8Update(uint8_t crc, uint8_t data)
{
    uint8_t bit;

        {
            crc = (uint8_t)((crc << 1U) ^ 0x07U);

        else
        {
            crc <<= 1U;
        }
    }
    return crc;
}

                               uint8_t sequence,
                               const uint8_t *payload,

{

    uint8_t crc = 0U;
    uint8_t index;

    if ((length > K230_LINK_MAX_PAYLOAD_LENGTH) ||
        ((length > 0U) && (payload == NULL)))
    {
        return;
    }

    frame[0] = K230_LINK_FRAME_MAGIC_0;
    frame[1] = K230_LINK_FRAME_MAGIC_1;
    frame[2] = K230_LINK_FRAME_VERSION;
    frame[3] = type;
    frame[4] = sequence;
    frame[5] = length;

    {
        crc = K230Link_Crc8Update(crc, frame[index]);
    }
    for (index = 0U; index < length; index++)

        crc = K230Link_Crc8Update(crc, payload[index]);

    Serial2_SendArray(frame, (uint16_t)(7U + length));
}

static void K230Link_ParseByte(uint8_t data)
{
    switch (s_parser.state)
    {
        case K230_PARSER_MAGIC_0:
            if (data == K230_LINK_FRAME_MAGIC_0)
            {
                s_parser.state = K230_PARSER_MAGIC_1;
            }
            break;

        case K230_PARSER_MAGIC_1:
            if (data == K230_LINK_FRAME_MAGIC_1)
            {
                s_parser.state = K230_PARSER_VERSION;
            }
            else
            {
                s_parser.state = (data == K230_LINK_FRAME_MAGIC_0) ?
                    K230_PARSER_MAGIC_1 : K230_PARSER_MAGIC_0;
            }
            break;

        case K230_PARSER_VERSION:
            if (data != K230_LINK_FRAME_VERSION)
            {
                K230Link_ResetParser();
                break;
            }
            s_parser.crc = K230Link_Crc8Update(0U, data);
            s_parser.state = K230_PARSER_TYPE;
            break;

        case K230_PARSER_TYPE:
            s_parser.type = data;
            s_parser.crc = K230Link_Crc8Update(s_parser.crc, data);
            s_parser.state = K230_PARSER_SEQUENCE;
            break;

        case K230_PARSER_SEQUENCE:
            s_parser.sequence = data;
            s_parser.crc = K230Link_Crc8Update(s_parser.crc, data);
            s_parser.state = K230_PARSER_LENGTH;
            break;

        case K230_PARSER_LENGTH:
            if (data > K230_LINK_MAX_PAYLOAD_LENGTH)
            {
                K230Link_ResetParser();
                break;
            }
            s_parser.length = data;
            s_parser.payloadIndex = 0U;
            s_parser.crc = K230Link_Crc8Update(s_parser.crc, data);
            s_parser.state = (data == 0U) ?
                K230_PARSER_CRC : K230_PARSER_PAYLOAD;
            break;

        case K230_PARSER_PAYLOAD:
            s_parser.payload[s_parser.payloadIndex] = data;
            s_parser.payloadIndex++;
            s_parser.crc = K230Link_Crc8Update(s_parser.crc, data);
            if (s_parser.payloadIndex >= s_parser.length)
            {
                s_parser.state = K230_PARSER_CRC;
            }
            break;

        case K230_PARSER_CRC:
            if (data == s_parser.crc)
            {
                K230Link_HandleFrame();
            }
            K230Link_ResetParser();
            break;

        default:
            K230Link_ResetParser();
            break;
    }
}
  1. 第 160～164 行：每来一个字节，只根据当前 state 解释；MAGIC_0 状态只关心它是不是 AA。
  2. 第 165～169 行：读到 AA 后进入 MAGIC_1，其他噪声保持等待 AA。
  3. 第 171～180 行：第二字节是 55 才进入版本；若又来 AA，就仍把它当成“可能的新帧第一个 AA”，因此 AA AA 55 能从第二个 AA 正确同步。
  4. 第 183～191 行：版本不是 01 就整帧放弃；正确时用版本作为 CRC 第一个字节并进入 TYPE。
  5. 第 193～197 行：保存 TYPE、更新 CRC、进入 SEQUENCE。
  6. 第 199～203 行：保存 SEQ、更新 CRC、进入 LENGTH。
  7. 第 205～210 行：LEN 大于 32 会在写 payload 前复位，防止数组越界。
  8. 第 211～216 行：保存长度、把 payloadIndex 清零、把 LEN 纳入 CRC；LEN=0 直接等 CRC，否则进入 PAYLOAD。
  9. 第 218～225 行：每个负载字节先存后递增索引并更新 CRC；收够 LEN 个后转到 CRC 状态。
  10. 第 228～234 行：接收 CRC 等于累计值才 HandleFrame；无论对错都立刻 Reset，下一字节重新寻找 AA。
  11. 第 236～240 行：default 防御内存破坏或非法枚举值，同样安全复位。

难点拆解：状态机为什么比“搜索整帧字符串”可靠

输入：任意分块到达的二进制字节，可能一次只有一个，也可能 FIFO 中已有整帧，还可能混入噪声。

过程：状态机只保留必要上下文；AA55 定位、LEN 决定还收多少、CRC 决定是否交付。

输出：只有版本、长度和 CRC 都正确的完整帧才进入 HandleFrame。

边界：坏版本、超长 LEN、坏 CRC 或非法 state 都复位；连续 AA 的重同步逻辑避免错过紧随其后的合法帧头。

设计原因：UART 是无消息边界的字节流，不能假设一次 Read 就刚好得到一帧。

void K230Link_Update(uint8_t elapsedTicks)
{
    uint8_t data;

    while (Serial2_ReadByte(&data) != 0U)
    {
        K230Link_ParseByte(data);
    }

    if (s_readyAcknowledged == 0U)
    {
        uint16_t ticks = (uint16_t)s_readyRetryTicks + elapsedTicks;

        s_readyRetryTicks = (ticks >= K230_LINK_READY_RETRY_TICKS) ?
            K230_LINK_READY_RETRY_TICKS : (uint8_t)ticks;
        if (s_readyRetryTicks >= K230_LINK_READY_RETRY_TICKS)
        {
            K230Link_SendFrame(
                K230_LINK_MESSAGE_READY,
                s_readySequence,
                NULL,
                0U);
            s_readyRetryTicks = 0U;
        }
    }

    if (s_capturePending != 0U)
    {
        uint16_t ticks = (uint16_t)s_captureTimeoutTicks + elapsedTicks;

        if (ticks >= K230_LINK_CAPTURE_TIMEOUT_TICKS)
        {
            /* 超时按失败上报，index 固定为 0。 */
            s_capturePending = 0U;
            s_captureTimeoutTicks = 0U;
            s_captureAckOk = 0U;
            s_captureAckIndex = 0U;
            s_captureAckPending = 1U;
        }
        else
        {
            s_captureTimeoutTicks = (uint8_t)ticks;
        }
    }
}
  1. 第 242～245 行：初始化首先初始化 UART2 并把解析器放回等待 AA。
  2. 第 246～253 行：清目标、双向 ready 标志和发送序号。
  3. 第 254～255 行：本机 READY 使用 sequence 0，NextSequence 后下一个序号变 1；重试计数直接预装为 10，使第一次 Update 立即发送 READY，而不是先等 100 ms。
  4. 第 256～260 行：清除拍照等待和结果槽。
  5. 第 263～270 行：Update 先读空 UART2 软件缓冲，把所有字节依次送入帧状态机；这样 ACK 可在本拍先改变状态，再决定是否重试。
  6. 第 272～277 行：本机 READY 尚未被 ACK 时累计重试 tick，并饱和到 10，避免 uint8_t 回绕。
  7. 第 278～286 行：达到 10 就发送同一个 s_readySequence 的零负载 READY，并把计时清零；同一序号反复重试便于 ACK 精确匹配。
  8. 第 289～294 行：拍照 pending 时累计等待时间，达到 100 tick 判定 1 秒超时。
  9. 第 295～300 行：超时被转换成一条本地失败 ACK：ok=0、index=0、ackPending=1，所以上层不必另写一套超时读取接口。
  10. 第 302～305 行：未超时就保存累计值。

uint8_t K230Link_RequestCapture(uint8_t count)
{
    uint8_t payload;

    if ((count == 0U) || (count > K230_LINK_CAPTURE_MAX_COUNT))
    {
        return 0U;
    }
    if (K230Link_IsReady() == 0U)
    {
        return 0U;
    }
    if (s_capturePending != 0U)
    {
        return 0U;
    }

    payload = count;
    K230Link_SendFrame(
        K230_LINK_MESSAGE_CAPTURE,
        K230Link_NextSequence(),
        &payload,
        1U);

    s_capturePending = 1U;
    s_captureTimeoutTicks = 0U;
    return 1U;
}

    return s_capturePending;
}

uint8_t K230Link_PopCaptureAck(uint8_t *ok, uint16_t *index)
{
    if ((ok == NULL) || (index == NULL) || (s_captureAckPending == 0U))
    {
        return 0U;
    }

    *ok = s_captureAckOk;
    *index = s_captureAckIndex;
    s_captureAckPending = 0U;
    return 1U;
}

  2. 第 315～323 行：GetTarget 检查指针和是否曾收到目标，然后整结构体复制。它不清 s_hasTarget，因此是“最新快照”而非 Pop；Telemetry 可重复显示同一目标。
  3. 第 325～340 行：拍照请求拒绝 count=0、超过 20、未 ready 或已有 pending；当前只允许一个在途请求。
  4. 第 342～347 行：payload 仅一个 count 字节，TYPE=CAPTURE，SEQ 每次取新值。
  5. 第 349～351 行：发送后置 pending 并从 0 开始超时计时，返回 1 只代表请求被接受。
  6. 第 354～357 行：IsCapturePending 提供只读忙状态。
  7. 第 359～364 行：PopCaptureAck 需要两个有效输出指针和待消费结果；网络 ACK 与本地超时失败都走这里。

5. 难点函数拆解

volatile 能解决什么，不能解决什么

输入：UART ISR 更新 writeIndex，主循环更新 readIndex。

过程：volatile 强迫每次从内存重新读取共享索引，32 位 MSPM0 对齐的 uint32_t 读写可原子完成；每个执行方只写自己的索引。

输出：主循环能看见 ISR 新写入的字节，ISR 能看见消费者已释放的空间。

边界：volatile 不是互斥锁，不会让“多步读改写”自动原子化，也不能让多个生产者同时安全写同一索引。本设计安全性来自单生产者/单消费者结构。

设计原因：接收 ISR 极短，不在中断里解析文本、算 CRC、printf 或启动运动，降低漏字节和打断控制循环的风险。

READY 为什么要双向确认

输入：本机发送 READY(seq=A)，对端也发送 READY(seq=B)。

过程：本机收到对端 READY 后发 READY_ACK(payload=B)；收到对端 ACK 且 payload=A 后置 readyAcknowledged。

输出：peerReadyReceived 与 readyAcknowledged 同时为 1 才 IsReady。

边界：本机自己的 READY 每 100 ms 用同一 sequence 重试，直到匹配 ACK；READY_ACK 自身使用另一个发送 sequence。

设计原因：只收到对端报到不能证明对端也收到了本机报到，双向握手避免单向线路故障被误判为完整连接。

UART2 ISR → Serial2_ReadByte → K230Link_ParseByte → HandleFrame → Target/CaptureAck 槽

7. 常见误解与调试提示

“115200 就表示一拍能发很多数据。”115200 bit/s 在常见 8N1 下约 11520 byte/s。Serial1 发送又是逐字节阻塞；高频长遥测会占用主循环时间，所以 Telemetry 会按字段长度限制安全频率。

“缓冲满了程序会报错。”当前 ISR 不报错，而是移动 readIndex 丢弃最旧字节。文本命令可能变成格式错误，二进制状态机则会等下一个 AA55 重新同步。调试时应观察接收速率和主循环是否被长时间阻塞。

“CRC 正确就代表消息可信。”CRC 只检测偶发传输错误，不是加密、身份认证或防重放。协议仍默认 UART2 对端就是可信 K230。

“SEQ 会阻止所有旧应答。”READY_ACK 会核对被确认的 sequence；TARGET 保存 sequence 供观察。但 CAPTURE_ACK 当前只检查 type、length 和 capturePending，没有核对请求 sequence。协议假设链路有序且旧 ACK 不会跨到下一请求。

“最新 TARGET 会自动过期。”不会。s_hasTarget 一旦置 1 就持续返回最后快照，除非重新初始化。需要识别失联/陈旧目标时，可由上层结合 sequence 变化或增加目标超时。

“输入 L--100 一定报错。”当前符号规则在首个数字前允许多次 +/−，最后一个符号生效。网页应发送规范格式；若要严格语法，可增加 hasSign 标志拒绝第二个符号。

修改联动清单

  • 改 UART 波特率或引脚：同步 main.syscfg、外部网页/蓝牙/K230 设置，并确认 Serial_Init 的 IOMUX 宏仍指向正确 RX。
  • 改缓冲容量：保持宏为非零；取模可支持任意容量，但 2 的幂通常更容易让编译器优化。还要评估静态 RAM 占用。
  • 增加文本命令：同时修改 IsCommand、ExecuteCommand、READY 帮助文本、网页端命令说明和范围测试。
  • 改变 100 Hz App 节拍：按真实毫秒重新计算 BLUETOOTH_COMMAND_IDLE_TICKS、READY_RETRY_TICKS 和 CAPTURE_TIMEOUT_TICKS。
  • 改变 K230 帧字段：MSPM0 与 uart_io.py 必须同步 VERSION、TYPE、LEN、字节序和 CRC 覆盖范围；只改一边会导致全部帧被丢弃。
  • 允许多个拍照请求或任务信号排队：单槽变量必须改成队列，并明确溢出策略、sequence 匹配和取消规则。
  • 提高遥测频率：不仅看 UART 带宽，还要看阻塞 printf 对 App_Update、MotionManager 和 K230 解析延迟的影响。

8. 学完自测

为什么 ISR 只搬字节，不直接解析命令？

ISR 应尽快排空硬件 FIFO并退出。文本解析、CRC、printf 和运动调用耗时且分支多，放主循环可减少中断占用和漏字节风险。

writeIndex=1100、readIndex=1000、容量 1024 时有多少可读字节，下一读槽是什么？

Available=100；真实槽位是 1000%1024=1000。逻辑索引不必小于数组容量。

L100R-50 没有分隔符，为什么仍能执行两条命令？

读到 R 时 StartCommand 发现已有 L100，先执行它，再建立 R；后续 -50 由分隔符或空闲超时执行。

C0 与 C5 同一拍到达，为什么停车不会丢？

C0 会覆盖单槽并锁住；只要它尚未 Pop，普通 C5 发布会被拒绝并返回 ERR STOP PENDING。

K230 CRC 覆盖哪些字节？

从 VERSION、TYPE、SEQ、LEN 到全部 PAYLOAD；不覆盖 AA55 MAGIC，也不覆盖 CRC 字节本身。

offsetX 的两个字节 0x9C、0xFF 还原为何值？

小端先拼成 uint16_t 0xFF9C，再转 int16_t，补码值为 -100。

为什么 K230Link_IsReady 要两个标志同时为真？

peerReadyReceived 证明本机收到对端，readyAcknowledged 证明对端收到本机；两者同时成立才确认双向链路。

P5 返回 OK CAP REQ 5 是否表示拍照成功？

不表示，只表示请求已发出。之后收到 CAPTURE_ACK 才输出 OK CAP index；1 秒超时或失败会输出 ERR CAP FAIL。
