08　通用任务状态机引擎：用静态状态图编排整车动作

先看它在整车中的位置

一句话职责
Mission 不知道“25H 应该走几毫米”，它只校验并执行 Accomplish 提供的静态状态图：进入状态、调用动作更新、按顺序检查转换、退出旧状态，再进入新状态。
上游输入
main 选择的一张 Mission_GraphDefinition_t 静态图，以及 App 每个有效节拍提供的 App_UpdateContext_t 时间、按键和蓝牙事件快照。
下游输出
当前状态编号、状态内经过时间、动作 RUNNING/FINISHED/ERROR、Mission 总体状态；转换时调用 Accomplish 回调并统一停止 MotionManager。
依赖模块
Application/Core/App 的更新上下文，MotionManager 的动作状态与停止接口，以及 Accomplish 提供的回调、状态表、转换表。
主要调用者
main.c 开机调用 Mission_Init，每次 App_Update 返回有效节拍后调用 Mission_Update。
总体位置
main/App → Mission 引擎 → Accomplish 状态回调 → MotionManager → 直行/巡线/转向/刹车。

初学者需要先知道

有限状态机：系统任一时刻只处在有限个状态中的一个。每个状态有进入、持续更新、退出三个阶段，并通过转换跳到另一个状态。

枚举：给整数取有意义的名字。写 MISSION_ACTION_RUNNING 比写裸数字 0 更不容易把含义弄错。

结构体：把一组相关字段装成一个类型。转换结构体像一条路线说明，状态结构体像一个站点说明。

函数指针：保存函数地址的变量。Mission 不直接知道题目函数名，而是通过 onEnter/onUpdate/onExit/condition 指针调用 Accomplish 提供的函数。

回调：把函数交给另一个模块，在合适时机由那个模块反过来调用。这里是题目把回调交给 Mission 引擎。

静态图定义：编译后长期存在、只描述“有哪些状态和边”的只读表。它不记录现在走到哪里。

运行时游标：当前状态、该状态经过时间、当前动作结果。它像地图上的“你在这里”标记，与地图本身分开。

只读指针：const Mission_GraphDefinition_t * 表示 Mission 只读取图，不修改题目静态表；图本身必须一直有效。

打断转换：动作仍 RUNNING 时，由传感器、按键或信号提前切换；只有状态允许 interruptible 才检查。

正常转换：动作 FINISHED 后检查的转换，例如直线走完后进入刹车。

短路求值：&& 左边为假时不再算右边，|| 左边为真时不再算右边；大量空指针防护依赖它。

单拍事件：只在当前 App 更新上下文中有效的信号。本拍没有条件接收，就不会由 Mission 自动缓存到以后。

Application/Mission/Mission.c（1/4）：静态图的结构合法性校验

#include "Application/Mission/Mission.h"
#include "Application/Control/MotionManager.h"
#include <stddef.h>

static Mission_Runtime_t s_runtime;
static Mission_Status_t s_status = MISSION_STATUS_UNINITIALIZED;
static const Mission_GraphDefinition_t *s_graph;
static uint8_t s_graphValid;
static uint8_t s_stateActive;

static uint8_t Mission_GraphIsValid(
    const Mission_GraphDefinition_t *graph)
{
    uint16_t stateIndex;

    if ((graph == NULL) || (graph->states == NULL) ||
        (graph->stateCount == 0U) ||
        (graph->startState >= graph->stateCount) ||
        (graph->errorState >= graph->stateCount))
    {
        return 0U;
    }

    for (stateIndex = 0U; stateIndex < graph->stateCount; stateIndex++)
    {
        const Mission_StateDefinition_t *state =
            &graph->states[stateIndex];
        uint8_t transitionIndex;

        if ((state->transitionCount != 0U) &&
            (state->transitions == NULL))
        {
            return 0U;
        }

        for (transitionIndex = 0U;
             transitionIndex < state->transitionCount;
             transitionIndex++)
        {
            const Mission_Transition_t *transition =
                &state->transitions[transitionIndex];

            if ((transition->targetState >= graph->stateCount) ||
                ((transition->type != MISSION_TRANSITION_NORMAL) &&
                 (transition->type != MISSION_TRANSITION_INTERRUPT)))
            {
                return 0U;
            }
        }
    }

    return 1U;
}
  1. 第 1～3 行：引入自身类型、运动管理器和 NULL 定义。
  2. 第 5～6 行：运行时游标与总体状态是引擎私有全局；状态显式从 UNINITIALIZED 开始。
  3. 第 7～9 行：s_graph 只保存题目静态图地址；另存校验结果和“当前状态是否已进入且尚未退出”。
  4. 第 11～14 行：图校验是私有函数，接收只读图指针并准备遍历状态数组的 16 位下标。
  5. 第 16～19 行：第一层防护：图和 states 都不能空，状态数不能为 0，起点和错误状态都必须落在数组范围内。
  6. 第 20～22 行：任一基础条件失败就返回无效。
  7. 第 24～28 行：逐个状态取只读指针，并准备遍历该状态转换表的 8 位下标。
  8. 第 30～34 行：若 transitionCount 非 0，就必须提供 transitions 地址；否则循环会解引用空指针。
  9. 第 36～39 行：按数组下标 0、1、2……遍历全部转换。这个声明顺序稍后就是同类型转换的优先顺序。
  10. 第 40～41 行：取出当前转换项的只读地址。
  11. 第 43～45 行：目标状态必须在范围内，类型必须恰好是 NORMAL 或 INTERRUPT。
  12. 第 46～49 行：发现非法转换立即失败；否则继续检查下一项和下一状态。
  13. 第 50 行：结束状态遍历。
  14. 第 52～53 行：所有结构检查通过才返回有效。

Application/Mission/Mission.c（2/4）：退出、进入与统一错误状态

static void Mission_CallExit(Mission_ExitReason_t reason)
{
    const Mission_StateDefinition_t *state;

    if ((s_stateActive == 0U) || (s_graph == NULL) ||
        (s_runtime.currentState >= s_graph->stateCount))
    {
        return;
    }

    state = &s_graph->states[s_runtime.currentState];
    if (state->onExit != NULL)
    {
        state->onExit(reason);
    }
    s_stateActive = 0U;
}

static uint8_t Mission_EnterState(uint16_t stateId)
{
    const Mission_StateDefinition_t *state;

    if ((s_graphValid == 0U) || (s_graph == NULL) ||
        (stateId >= s_graph->stateCount))
    {
        return 0U;
    }

    state = &s_graph->states[stateId];
    s_runtime.currentState = stateId;
    s_runtime.stateElapsedSeconds = 0.0f;
    s_runtime.actionStatus = (state->onUpdate == NULL) ?
        MISSION_ACTION_FINISHED : MISSION_ACTION_RUNNING;
    s_stateActive = 1U;

    if ((state->onEnter != NULL) &&
        (state->onEnter() != MISSION_CALLBACK_OK))
    {
        s_runtime.actionStatus = MISSION_ACTION_ERROR;
        return 0U;
    }

    s_status = (s_runtime.actionStatus == MISSION_ACTION_FINISHED) ?
        MISSION_STATUS_COMPLETED : MISSION_STATUS_RUNNING;
    return 1U;
}

static void Mission_SetError(void)
{
    const Mission_StateDefinition_t *errorState;

    Mission_CallExit(MISSION_EXIT_ERROR);
    MotionManager_Stop();

    if ((s_graphValid == 0U) || (s_graph == NULL) ||
        (s_graph->errorState >= s_graph->stateCount))
    {
        s_runtime.currentState = 0U;
        s_runtime.stateElapsedSeconds = 0.0f;
        s_runtime.actionStatus = MISSION_ACTION_ERROR;
        s_stateActive = 0U;
        s_status = MISSION_STATUS_ERROR;
        return;
    }

    s_runtime.currentState = s_graph->errorState;
    s_runtime.stateElapsedSeconds = 0.0f;
    s_runtime.actionStatus = MISSION_ACTION_ERROR;
    s_stateActive = 1U;
    s_status = MISSION_STATUS_ERROR;

    errorState = &s_graph->states[s_graph->errorState];
    if (errorState->onEnter != NULL)
    {
        (void)errorState->onEnter();
    }
}

  2. 第 59～63 行：只有状态确实 active、图存在且当前下标有效才允许退出；重复退出会被挡住。
  3. 第 65～69 行：找到当前状态；若 onExit 非空就把 completed/interrupted/stopped/error 原因传给题目回调。
  4. 第 70～71 行：无论有没有退出回调，都把 active 清 0，表示生命周期已关闭。
  5. 第 73～80 行：进入函数先检查图已校验、图指针存在且目标下标合法。
  6. 第 81 行：结束非法目标的提前返回分支。
  7. 第 83～85 行：选中目标状态，移动运行时游标，并把该状态计时归零。
  8. 第 86～88 行：没有 onUpdate 的状态被预设为立即 FINISHED；有 onUpdate 才从 RUNNING 开始。随后先标记 active。
  9. 第 90～95 行：若存在 onEnter 就调用一次；返回非 OK 时把动作标 ERROR 并报告进入失败。
  10. 第 97～100 行：进入成功后，根据动作是否已立即完成设置 Mission 总体 COMPLETED 或 RUNNING。
  11. 第 102～106 行：统一错误处理先对旧状态调用 ERROR 退出，再无条件停止 MotionManager。
  12. 第 107 行：执行统一停车，保证后续错误状态不会继承旧运动。
  13. 第 109～117 行：若图本身无效或错误状态不合法，使用安全兜底运行时：状态 0、计时 0、动作 ERROR、无 active、总体 ERROR。
  14. 第 118 行：结束无有效错误状态时的兜底分支。
  15. 第 120～124 行：有效图则把运行时游标直接放到 graph 指定的 errorState，动作与总体都标 ERROR。
  16. 第 126～130 行：若错误状态有 onEnter，就调用一次，但用 (void) 明确忽略返回值；错误状态不会因自身进入失败再递归报错。
  17. 第 131 行：结束错误处理。

Application/Mission/Mission.c（3/4）：回起点、按表转换与初始化

static void Mission_ResetToStart(void)
{
    Mission_CallExit(MISSION_EXIT_STOPPED);
    MotionManager_Stop();

    if (Mission_EnterState(s_graph->startState) == 0U)
    {
        Mission_SetError();
    }
}

static uint8_t Mission_TryTransition(
    Mission_TransitionType_t transitionType,
    const App_UpdateContext_t *updateContext)
{
    const Mission_StateDefinition_t *state =
        &s_graph->states[s_runtime.currentState];
    uint8_t transitionIndex;

    for (transitionIndex = 0U;
         transitionIndex < state->transitionCount;
         transitionIndex++)
    {
        const Mission_Transition_t *transition =
            &state->transitions[transitionIndex];
        Mission_ExitReason_t exitReason;

        if (transition->type != transitionType)
        {
            continue;
        }
        if ((transition->condition != NULL) &&
            (transition->condition(&s_runtime, updateContext) == 0U))
        {
            continue;
        }

        exitReason = (transitionType == MISSION_TRANSITION_INTERRUPT) ?
            MISSION_EXIT_INTERRUPTED : MISSION_EXIT_COMPLETED;
        Mission_CallExit(exitReason);
        MotionManager_Stop();
        if (Mission_EnterState(transition->targetState) == 0U)
        {
            Mission_SetError();
        }
        return 1U;
    }

    return 0U;
}

void Mission_Init(const Mission_GraphDefinition_t *graph)
{
    s_graph = graph;
    s_graphValid = Mission_GraphIsValid(graph);
    s_runtime.currentState = 0U;
    s_runtime.stateElapsedSeconds = 0.0f;
    s_runtime.actionStatus = MISSION_ACTION_ERROR;
    s_status = MISSION_STATUS_UNINITIALIZED;
    s_stateActive = 0U;

    if (s_graphValid == 0U)
    {
        MotionManager_Stop();
        s_status = MISSION_STATUS_ERROR;
        return;
    }

    if (Mission_EnterState(s_graph->startState) == 0U)
    {
        Mission_SetError();
    }
}
  1. 第 133～137 行：回起点先以 STOPPED 原因退出当前状态，并停止任何运动。
  2. 第 138～142 行：再进入图定义的 startState；起点 onEnter 失败就转统一错误状态。
  3. 第 144～150 行：转换尝试接收本次只检查的类型和 App 快照，并取得当前状态及遍历下标。
  4. 第 152～157 行：严格按转换数组声明顺序逐项读取，这就是“第一条满足者优先”的实现。
  5. 第 158 行：为命中的转换准备正确退出原因。
  6. 第 160～163 行：不是本次请求类型就 continue，跳过它但保留数组内后续同类型项的顺序。
  7. 第 164～168 行：condition 非空时调用；返回 0 表示不成立并继续。condition 为 NULL 时整段为假，因此无条件通过。
  8. 第 170～173 行：打断边使用 INTERRUPTED，正常边使用 COMPLETED；随后先退出旧状态，再统一停止旧运动。
  9. 第 174～177 行：进入目标状态；失败时立即进入统一错误处理。
  10. 第 178～182 行：只要执行了一条转换就马上返回 1；整张表没有命中才返回 0。因此单次调用最多转换一次。
  11. 第 184～187 行：初始化保存图地址并立即做结构校验；Mission 不复制图。
  12. 第 188～192 行：把运行时设置为安全默认：状态 0、计时 0、动作 ERROR、总体未初始化、当前状态未激活。
  13. 第 194～199 行：图无效时停车并把总体状态置 ERROR，不尝试访问 startState。
  14. 第 201～205 行：图有效时进入起点；起点进入失败则走统一错误状态。

Application/Mission/Mission.c（4/4）：每拍推进、全局停止与上下文辅助

void Mission_Update(const App_UpdateContext_t *updateContext)
{
    const Mission_StateDefinition_t *state;

    if ((updateContext == NULL) || (s_graphValid == 0U) ||
        (s_status == MISSION_STATUS_UNINITIALIZED))
    {
        return;
    }

    if (Mission_ContextHasBluetoothSignal(updateContext, 0U) != 0U)
    {
        Mission_ResetToStart();
        return;
    }

    if (s_status == MISSION_STATUS_ERROR)
    {
        return;
    }

    state = &s_graph->states[s_runtime.currentState];
    s_runtime.stateElapsedSeconds += updateContext->dt;

    if ((s_runtime.actionStatus == MISSION_ACTION_RUNNING) &&
        (state->onUpdate != NULL))
    {
        s_runtime.actionStatus = state->onUpdate(updateContext->dt);
    }

    if (((s_runtime.actionStatus != MISSION_ACTION_RUNNING) &&
         (s_runtime.actionStatus != MISSION_ACTION_FINISHED) &&
         (s_runtime.actionStatus != MISSION_ACTION_ERROR)) ||
        (s_runtime.actionStatus == MISSION_ACTION_ERROR) ||
        (MotionManager_GetError() != MOTION_MANAGER_ERROR_NONE))
    {
        Mission_SetError();
        return;
    }

    if (s_runtime.actionStatus == MISSION_ACTION_RUNNING)
    {
        s_status = MISSION_STATUS_RUNNING;
        if (state->interruptible != 0U)
        {
            (void)Mission_TryTransition(
                MISSION_TRANSITION_INTERRUPT, updateContext);
        }
        return;
    }

    s_status = MISSION_STATUS_COMPLETED;
    (void)Mission_TryTransition(MISSION_TRANSITION_NORMAL, updateContext);
}

void Mission_Stop(void)
{
    if ((s_graphValid == 0U) ||
        (s_status == MISSION_STATUS_UNINITIALIZED))
    {
        MotionManager_Stop();
        return;
    }

    Mission_ResetToStart();
}

Mission_Status_t Mission_GetStatus(void)
{
    return s_status;
}

const Mission_Runtime_t *Mission_GetRuntime(void)
{
    return &s_runtime;
}

Mission_ActionStatus_t Mission_GetMotionActionStatus(void)
{
    if (MotionManager_GetError() != MOTION_MANAGER_ERROR_NONE)
    {
        return MISSION_ACTION_ERROR;
    }
    if (MotionManager_IsFinished() != 0U)
    {
        return MISSION_ACTION_FINISHED;
    }
    if (MotionManager_IsBusy() != 0U)
    {
        return MISSION_ACTION_RUNNING;
    }
    return MISSION_ACTION_FINISHED;
}

uint8_t Mission_ContextHasBluetoothSignal(
    const App_UpdateContext_t *updateContext, uint8_t signal)
{
    if ((updateContext == NULL) ||
        (updateContext->hasBluetoothSignal == 0U))
    {
        return 0U;
    }

    return (updateContext->bluetoothSignal == signal) ? 1U : 0U;
}

难点函数拆解

难点一：静态图定义与运行时为什么必须分离

输入：Accomplish 提供的 states 数组、转换数组和 graph 描述；Mission 自己维护的 runtime。

过程：图定义只回答“可能有哪些状态和边”，运行时只回答“现在在哪、待了多久、动作怎样”。一张只读图可以由同一个通用引擎解释。

输出：换题目时只换 Accomplish...GetMissionGraph() 的返回图，不改 Mission 执行代码。

边界条件：Mission 只保存指针，因此 graph、states、transitions 必须具有静态存储期；不能在函数局部变量里创建后返回。

设计原因：数据描述与执行算法解耦，避免把 25E、25H、Test 的流程硬编码进公共模块。

难点二：四种函数指针回调在生命周期中的位置

输入：onEnter、onUpdate、onExit、condition 四类函数地址。

过程：进入状态只调用一次 onEnter；每拍且动作 RUNNING 时调用 onUpdate；准备离开时调用一次 onExit；只有扫描某条转换时才调用该条 condition。

输出：题目代码控制动作启动、动作状态和转换条件，却不直接控制引擎循环。

边界条件：签名必须完全匹配 typedef。onEnter 不接 dt，condition 不能返回动作三态，onExit 没有返回值。NULL 的含义也不同：空 onUpdate 是立即完成，空 condition 是无条件成立。

设计原因：固定签名让 Mission 能用同一套代码调用任意题目的函数。

难点三：转换表顺序、类型过滤和每拍最多一次转换

输入：当前状态的转换数组、本拍要检查的 NORMAL 或 INTERRUPT 类型、App 上下文。

过程：从数组第 0 项开始；类型不符跳过；条件非空且返回 0 也跳过；第一条通过者执行并立即 return 1。Mission_Update 的 RUNNING 分支调用一次 interrupt 扫描后立刻 return，FINISHED 分支只调用一次 normal 扫描。

输出：一拍最多走一条边，新状态即使没有 onUpdate 而立即完成，也要等下一拍才能继续走正常边。

边界条件：无条件边必须放在同类型表的最后。引擎没有检查数组是否“按 NORMAL/INTERRUPT 分组”，因为每次会过滤类型；不同类型之间不会在同一拍竞争。

设计原因：阻止一拍内连跳多个状态，避免多个 onEnter 连续启动/停止电机，也让传感器事件与时间推进更可预测。

难点四：RUNNING、FINISHED、ERROR 如何决定下一步

输入：题目 onUpdate 返回值，以及 MotionManager 的 error/busy/finished。

过程：非法枚举、动作 ERROR 或 Manager error 立即 SetError；RUNNING 只允许可打断状态扫描 INTERRUPT；FINISHED 把 Mission 标 COMPLETED并扫描 NORMAL。

输出：运动状态通常用 Mission_GetMotionActionStatus 适配 Manager；等待状态自行持续返回 RUNNING。

边界条件：适配函数必须先查 finished 再查 busy，因为 Straight CONTINUING 可二者同时为真。Manager IDLE 且无错误时兜底返回 FINISHED。

设计原因：把“动作执行结果”和“转换条件”分开：动作是否完成由状态回调回答，去哪个状态由转换表回答。

难点五：普通 interruptible 与全局 C0 为什么是两条通道

输入：状态定义的 interruptible、普通条件边，以及 App 快照中的蓝牙 signal 0。

过程：C0 在 Update 最前面直接 ResetToStart；普通打断只有 action RUNNING 且 interruptible 非 0 时才扫描。

输出：关键动作可以屏蔽普通题目切换，却无法屏蔽全局停车。

边界条件：C0 会退出当前状态、停止运动并重新调用 startState 的 onEnter。它不是简单把 mode 设 IDLE；若起点进入失败，仍会进入 errorState。更重要的是，所有合法任务图都必须把 startState 设计成不会自动启动运动的被动等待状态，否则 C0 刚停车就可能在同一拍因起点 onEnter 再次发出运动命令；当前图校验没有检查这条安全约束。

设计原因：安全停车权限不能被某道题的 interruptible 配置意外关闭。

难点六：错误状态不是普通状态

输入：进入回调失败、动作 ERROR、非法动作枚举或 MotionManager error。

过程：先用 ERROR 原因退出当前状态并停车；随后直接设置 runtime 到 errorState，调用它的 onEnter，但忽略返回值。Mission 总体保持 ERROR，后续普通 Update 不执行 errorState 的 onUpdate。

输出：车辆安全停止，调试界面能看到固定错误状态；C0 仍可尝试回起点。

边界条件：errorState 的 onEnter 应简单、不可再依赖可能失败的运动启动。由于返回值被忽略，它更适合蜂鸣、标志或静态错误显示。

设计原因：错误处理必须有终点，不能因“错误状态进入也失败”无限递归。

常见误解与调试提示

误解：图校验通过就代表流程一定正确

校验只保证指针、数量、目标下标和转换类型不会明显越界。它不检查状态是否可达、无条件边是否遮住后续边、条件是否永远为假、图指针生命周期是否正确。

误解：interruptible=0 会阻止所有外部事件

它只禁止普通 INTERRUPT 转换。蓝牙 C0 在更前面处理，任何状态包括 ERROR 都无法屏蔽。

误解：空 onUpdate 表示“什么都不做，继续等”

源码把空 onUpdate 直接标为 FINISHED。等待按键或蓝牙的状态必须提供一个每拍返回 RUNNING 的更新回调。

误解：onEnter 可以循环等待动作完成

onEnter 只应启动一次动作并迅速返回。阻塞会冻结 App、MotionManager 和传感器更新；完成等待应交给后续 onUpdate。

误解：条件回调里可以顺便启动电机

条件可能每拍、按多条边反复执行，应只回答真假。电机动作只能在目标状态 onEnter 中启动，所有权与生命周期才清晰。

误解：文档写“当前 25H”，固件就一定在跑 25H

文档可能滞后；当前 main 实际加载 AccomplishTest。判断当前行为应查看编译中的 main.c、工程文件和最终构建，而不是只看示例章节。

修改结构或规则会牵动哪里

  • 改变回调 typedef 会要求所有 Accomplish 回调同步改签名，否则函数指针类型不匹配。
  • 给 Mission_StateDefinition_t 增加字段时，所有静态状态表的指定初始化都要审查默认值与含义。
  • 改变转换扫描顺序会改变同拍多个条件同时成立时的实际路径，所有题目的转换表优先级都可能变化。
  • 允许一拍多次转换会导致连续 onExit/Stop/onEnter，可能让单拍事件被多个状态重复观察，也会增加电机切换风险。
  • 改变空 onUpdate 的语义，会影响所有等待状态、错误状态和只靠正常边自动前进的状态。
  • 改变 C0 处理位置可能使 ERROR 或 non-interruptible 状态无法停车，属于安全语义变更。
  • 改变 Mission_GetMotionActionStatus 的 finished/busy 检查顺序，会破坏 Straight CONTINUING 的分段完成语义。
  • 修改 App 与 Mission 的调用顺序，会决定条件和动作看到的是本拍更新前还是更新后的传感器/运动状态。

学完自测

1. 静态图和 Mission_Runtime 分别保存什么？

静态图保存所有状态、转换、起点和错误状态；Runtime 只保存当前状态编号、进入该状态后的秒数和当前动作状态。

2. 一个状态 onUpdate 为 NULL，进入后 actionStatus 是什么？

立即设为 MISSION_ACTION_FINISHED，Mission 总体先变 COMPLETED；下一次 Update 会检查正常转换。

3. 同一状态两条 INTERRUPT 条件同时为真，哪条生效？一拍会跳几次？

转换数组中靠前的同类型条目生效；命中后立即返回，所以一拍最多一次转换。

4. interruptible=0 时收到 C0 会怎样？

仍会以 STOPPED 原因退出、停止 MotionManager 并重新进入 startState；C0 检查早于 interruptible。

5. 为什么 Mission_GetMotionActionStatus 先检查 finished 再检查 busy？

Straight 的 CONTINUING 状态会同时 finished=1、busy=1；先看 finished 才能让 Mission 在分段距离到达后转换到下一状态。
