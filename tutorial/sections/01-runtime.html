模块 01：从 main 到 100Hz 主循环——启动、节拍与 App 总调度

这一节先不钻进巡线、转向或通信协议内部，而是回答最关键的问题：程序从哪里开始，为什么大约每 10 ms 工作一次，所有子系统按什么顺序协作，以及数据怎样从 App 交给 Mission。

1. 先看它在整车中的位置

一句话职责
建立整车运行环境，收集 100Hz 系统节拍，把传感、通信、运动、提示和显示串成一次完整更新，再把本周期事件交给任务状态机。
上游输入
SysTick 中断产生的节拍、GPIO 按键状态、UART/K230 数据、陀螺仪和编码器数据、运动模块错误。
下游输出
App_UpdateContext_t 时间/事件快照、运动更新、急停、蜂鸣、遥测、OLED 刷新，以及对 Mission_Update() 的调用许可。
依赖模块
SysConfig、Tick、Interrupt、板级驱动、通信、状态估计、MotionManager、调试显示和遥测。
调用者
main() 调用 App_Init() 和 App_Update()；硬件在固定时刻调用 SysTick_Handler()。
总体位置
main → App → Mission → MotionManager → Hardware 是命令与所有权关系：Mission 通过 MotionManager 请求动作。单个有效节拍的真实执行顺序是 App_Update() 先更新状态、通信和 MotionManager，返回后 main 才调用 Mission_Update() 决定下一动作。App 同时向通信、蜂鸣、遥测和显示支线分发时间。

2. 初学者需要先知道

  • 包含：让当前源文件先看见另一个头文件里的类型、宏和函数声明，像在写正文前拿到“名词表”。它不会自动调用任何函数。
  • 程序入口：单片机完成更底层的复位启动后进入 main()，这里是本项目 C 代码的总入口。
  • 中断：硬件事件到来时暂时打断主程序，先执行一个很短的处理函数，再回到原位置。SysTick 每 10 ms 触发一次中断。
  • 中断服务函数：专门响应中断的函数；本模块的 SysTick_Handler() 只负责累计节拍，不在中断里运行复杂业务。
  • 结构体：把多种相关数据装进一个有名字的“表格”。App_UpdateContext_t 就是一张交给 Mission 的本周期快照。
  • 指针：保存内存地址的变量。&updateContext 取得结构体地址，让 App_Update() 能直接填写 main 中的那一份结构体。
  • 空指针：表示“没有指向有效对象”。函数先检查它，避免通过无效地址写内存。
  • 静态/文件私有：在文件作用域使用时，名字只对本 .c 文件可见，同时变量在整个运行期间一直存在。
  • 易变对象：提醒编译器该变量可能被中断异步修改，每次都要真正读写内存；它不等于线程安全，也不能代替关中断保护。
  • 位掩码：用一个整数的不同二进制位表示多个开关。本项目用 bit0～bit3 表示四个按键。
  • 边沿：状态从“未按”变化到“按下”的那一瞬间；与一直保持为 1 的“电平状态”不同。
  • 时间增量：两次有效更新之间实际过去的秒数，控制和积分算法不能假定主循环永远只漏过 0 个节拍。
  • 100U 的 U 表示无符号整数，0.01f 的 f 表示单精度浮点数；这些后缀让常量类型与变量/接口更一致。

3. 文件导读

文件为什么存在公开与私有分工
main.c规定启动顺序和永不退出的顶层循环。只做最高层编排，不实现具体驱动。
System/Tick.h/.c把硬件 SysTick 中断变成主循环可取走的“累计节拍数”。头文件公开频率、时间步长和查询函数；源文件隐藏计数器并实现 ISR。
System/Interrupt.h/.c把全局开/关中断包装成语义清楚的项目接口。公开两个操作；实现调用 ARM 内核指令。
System/Delay.h/.c提供微秒、毫秒、秒级阻塞等待。它是工具模块，不负责 100Hz 主循环调度。
Application/Core/App.h/.c初始化所有基础模块，并在每次有效节拍中按固定顺序调度它们。头文件公开上下文结构和两个入口；源文件保存“上一拍”状态与具体流水线。

先纠正文档差异：README 的旧说明仍写着默认装载 Accomplish25H_GetMissionGraph()，甚至说 Test 只应临时启用；但当前真正参与编译的 main.c 第 1 行包含 Accomplish/Test.h，第 11 行调用 AccomplishTest_GetMissionGraph()。因此当前固件实际运行的是刹车测试任务图，而不是 README 描述的 25H。判断“现在烧进去什么程序”时，应以当前源码和构建配置为准。

4. 逐行讲解

4.1 main.c：启动顺序和永久循环

#include "Accomplish/Test.h"
#include "Application/Core/App.h"
#include "Application/Mission/Mission.h"
#include "System/Interrupt.h"

int main(void)
{
    App_UpdateContext_t updateContext;

    App_Init();
    Mission_Init(AccomplishTest_GetMissionGraph());
    Interrupt_Enable();
  1. 第 1 行：取得刹车测试任务图的函数声明。这一行已经是“当前加载 Test”的第一份直接证据。
  2. 第 2～4 行：分别取得 App 总调度、Mission 状态机和全局中断接口。main 只依赖公开头文件，不需要知道它们的内部变量。
  3. 第 6～7 行：int main(void) 表示函数不接收参数、返回整数；在单片机中正常情况下不会走到返回。花括号圈出函数体。
  4. 第 8 行：在 main 的栈空间创建一份上下文结构体。此时字段还没有有效业务含义，稍后由 App_Update() 填写。
  5. 第 10 行：先初始化硬件、状态估计、通信、运动管理和显示等基础设施。
  6. 第 11 行：先调用 AccomplishTest_GetMissionGraph() 取得测试任务图地址，再把它交给 Mission 初始化。这里不是 25H。
  7. 第 12 行：最后统一打开全局硬件中断。这样能避免初始化到一半时，串口、编码器或 SysTick 的 ISR 就开始使用尚未准备好的状态。
    for (;;)
    {
        if (App_Update(&updateContext) != 0U)
        {
            Mission_Update(&updateContext);
        }
    }
}
  1. 第 14～15 行：for (;;) 省略初始化、条件和步进，因此条件永远不会变成假；这是单片机常见的无限循环，固件会一直值守。
  2. 第 16 行：& 取得 updateContext 地址。App 若完成一次有效节拍更新便返回非零；若无节拍或参数无效则返回 0。
  3. 第 17～19 行：只有 App 已经更新好时间、按键和通信事件后，Mission 才读取同一份上下文。这样 Mission 不会在没有新时间片时空转。
  4. 第 20～21 行：闭合循环和 main。源码没有 return，因为设计目标就是永久运行。

4.2 System/Tick.h：100Hz 时间契约

#ifndef TICK_H
#define TICK_H

#include <stdint.h>

#define TICK_HZ 100U
#define TICK_DT 0.01f

void Tick_Init(void);
uint8_t Tick_Poll(void);
uint8_t Tick_PollCount(void);

#endif
  1. 第 1～2、13 行：这是头文件保护。第一次包含时定义 TICK_H，以后再次包含便跳过正文，避免重复声明。
  2. 第 4 行：引入固定宽度整数类型；uint8_t 是 8 位无符号整数，范围 0～255。
  3. 第 6 行：声明系统节拍频率为每秒 100 次。对应周期是 1 / 100 = 0.01 秒，即 10 ms。
  4. 第 7 行：直接给出一个节拍对应的秒数，供浮点时间算法使用。TICK_HZ 与 TICK_DT 必须保持互为倒数。
  5. 第 9 行：声明计数器和 SysTick 当前值的复位函数。
  6. 第 10 行：声明布尔式查询：只回答“有没有至少一拍”，会舍弃具体累计数量。
  7. 第 11 行：声明计数式查询：原子取出从上次查询以来累计的节拍数；当前 App 使用的是它。

硬件配置还必须与这两个宏一致。当前 main.syscfg 把 SysTick 周期设为 320000；CPU 为 32MHz，因此 32,000,000 / 320,000 = 100Hz。宏只是软件约定，真正的中断速度来自硬件配置。

4.3 System/Tick.c：中断累计与安全取走

#include "System/Tick.h"
#include "ti_msp_dl_config.h"

static volatile uint8_t s_tickCount;

void Tick_Init(void)
{
    uint32_t primask = __get_PRIMASK();
    __disable_irq();
    s_tickCount = 0U;
    SysTick->VAL = 0U;
    SCB->ICSR = SCB_ICSR_PENDSTCLR_Msk;
    __set_PRIMASK(primask);
}
  1. 第 1～2 行：先取得本模块接口，再取得 TI 生成的芯片寄存器、时钟宏和 ARM 内核函数声明。
  2. 第 4 行：s_tickCount 是文件私有、全程存在的 8 位计数器；volatile 是因为主循环和中断都会访问它。前缀 s_ 是项目对静态变量的命名习惯。
  3. 第 6～7 行：开始节拍复位函数。
  4. 第 8 行：读取 全局中断屏蔽状态。它记录调用本函数前中断本来是开还是关。
  5. 第 9 行：临时屏蔽普通硬件中断，进入很短的临界区，避免复位进行一半时 SysTick ISR 插入。
  6. 第 10 行：清空软件累计值。
  7. 第 11 行：向 SysTick 当前计数值寄存器写 0，让硬件计时从一个干净起点重新装载。
  8. 第 12 行：向系统控制块的中断状态寄存器写“清除 SysTick 挂起位”掩码，丢掉复位前已经等待处理的一次 SysTick。
  9. 第 13～14 行：恢复进入函数前的 PRIMASK，而不是无条件开中断。因此 App 初始化期间调用 Tick_Init() 后，中断仍能保持关闭。
uint8_t Tick_PollCount(void)
{
    uint8_t count;
    uint32_t primask = __get_PRIMASK();
    __disable_irq();
    count = s_tickCount;
    s_tickCount = 0U;
    __set_PRIMASK(primask);
    return count;
}

uint8_t Tick_Poll(void) { return (Tick_PollCount() != 0U) ? 1U : 0U; }
  1. 第 16～18 行：函数返回累计拍数，并准备局部变量 count 保存快照。
  2. 第 19～20 行：保存原中断状态后进入临界区。
  3. 第 21～22 行：先复制、再清零，语义像“取走邮箱中的全部节拍”。这两行必须不可被 SysTick ISR 插入；否则 ISR 在复制和清零之间加的一拍会被错误清掉。
  4. 第 23～25 行：恢复原中断状态并返回快照。临界区只包含必要操作，尽量减少中断延迟。
  5. 第 27 行：三目运算符 条件 ? 值1 : 值2 把累计数转换成 1 或 0。它内部也会清空计数；当前 App 不用该接口，因为 App 需要知道漏过了几拍。
void SysTick_Handler(void)
{
    if (s_tickCount < 255U) s_tickCount++;
}
  1. 第 29～30 行：函数名由芯片启动文件/中断向量表约定；SysTick 到期时硬件自动跳到这里。
  2. 第 31 行：若没到 255 就加 1；到 255 后保持不变，称为饱和计数。这样不会从 255 溢出回 0，把“积压很多拍”误报成“没有拍”。
  3. 第 32 行：ISR 到此立即结束。中断只记账，复杂更新留给主循环，降低中断占用时间。

4.4 System/Interrupt.h/.c：统一开关全局中断

#ifndef SYSTEM_INTERRUPT_H
#define SYSTEM_INTERRUPT_H

/* 所有驱动和 Mission 初始化完成后，统一开启全局硬件中断。 */
void Interrupt_Enable(void);
void Interrupt_Disable(void);

#endif
  1. 第 1～2、8 行：头文件保护。
  2. 第 4 行：把项目级规则写在公开接口旁：初始化完成前不允许全局中断介入。
  3. 第 5～6 行：分别公开开、关中断函数。它们控制的是 CPU 全局接收，而不是删除某个外设的配置。
#include "System/Interrupt.h"
#include "ti_msp_dl_config.h"

void Interrupt_Enable(void)
{
    __enable_irq();
}

void Interrupt_Disable(void)
{
    __disable_irq();
}
  1. 第 1～2 行：取得项目声明和 ARM/TI 内核指令声明。
  2. 第 4～7 行：Interrupt_Enable() 用 __enable_irq() 清除全局屏蔽，使已配置且获准的外设中断能够响应。
  3. 第 9～12 行：Interrupt_Disable() 设置全局屏蔽。已经挂起的中断通常不会因此被删除，重新开启后仍可能立刻响应。

4.5 System/Delay.h/.c：阻塞延时不是调度节拍

#ifndef DELAY_H
#define DELAY_H

#include <stdint.h>

void Delay_us(uint32_t us);
void Delay_ms(uint32_t ms);
void Delay_s(uint32_t s);

#endif
  1. 第 1～4、10 行：头文件保护并引入 uint32_t。
  2. 第 6～8 行：公开微秒、毫秒、秒三个等待接口；参数是 32 位无符号数，不能表达负时间。
#include "System/Delay.h"
#include "ti_msp_dl_config.h"

void Delay_us(uint32_t us)
{
    const uint32_t cyclesPerUs = CPUCLK_FREQ / 1000000U;
    while (us-- > 0U) delay_cycles(cyclesPerUs);
}

void Delay_ms(uint32_t ms)
{
    while (ms-- > 0U) Delay_us(1000U);
}

void Delay_s(uint32_t s)
{
    while (s-- > 0U) Delay_ms(1000U);
}
  1. 第 1～2 行：取得函数声明，以及生成配置中的 CPUCLK_FREQ 与 delay_cycles()。
  2. 第 4～6 行：进入微秒延时，并计算 1 微秒含多少 CPU 周期；当前 32MHz 下为 32 个周期。const 表示此局部值计算后不再修改。
  3. 第 7 行：us-- 是先用旧值比较、再减 1；每个微秒执行一次忙等。传 0 时循环体不执行。CPU 在等待期间不能去做 App/Mission，所以它是阻塞延时。
  4. 第 10～13 行：毫秒函数把 1ms 拆成 1000 次 1μs 延时，并重复 ms 次。
  5. 第 15～18 行：秒函数再把 1s 拆成 1000 次 1ms 延时。层层复用简单，但长延时会长期占住主循环。

不要用 Delay_ms(10) 代替 100Hz 调度。Delay 是“站在原地等”，Tick 是“硬件定时打铃并累计迟到次数”。主循环若因串口或算法忙了 30ms，Tick 能报告约 3 拍；固定 Delay 无法补偿真实耗时。

4.6 Application/Core/App.h：交给 Mission 的快照

#ifndef APPLICATION_CORE_APP_H
#define APPLICATION_CORE_APP_H

#include <stdint.h>

/* 每次有效系统更新提供给 Mission 的只读事件和时间数据。 */
typedef struct
{
    uint8_t elapsedTicks;
    float dt;
    uint8_t pressedKeys;
    uint8_t pressedEdges;
    uint8_t hasBluetoothSignal;
    uint8_t bluetoothSignal;
} App_UpdateContext_t;

void App_Init(void);
uint8_t App_Update(App_UpdateContext_t *context);

#endif
  1. 第 1～4、20 行：头文件保护并引入固定宽度整数。
  2. 第 6 行：“只读”是使用约定：App 填写，Mission 接收 const App_UpdateContext_t * 后只读取。
  3. 第 7～8、15 行：typedef struct { ... } App_UpdateContext_t 定义一个匿名结构体，再给它取类型别名。以后可直接写 App_UpdateContext_t x;。
  4. 第 9 行：从上次成功查询后累计了几拍，1 通常表示 10ms，3 表示约 30ms。
  5. 第 10 行：同一时间用秒表示，计算式为 elapsedTicks × 0.01f。
  6. 第 11 行：当前仍处于按下状态的按键位图，适合“按住期间持续生效”。
  7. 第 12 行：本次新按下的按键位图，适合“一次按下只触发一次任务”。
  8. 第 13～14 行：前者说明本拍是否弹出了一条 UART1 任务信号，后者才是信号值。分成两个字段后，合法的信号 0 不会与“没有信号”混淆。
  9. 第 17 行：公开一次性初始化入口。
  10. 第 18 行：参数是可写结构体指针；返回 8 位值作为成功/无更新标志。

static void App_ReportMotionError(void)
{
    MotionManager_Error_t error = MotionManager_GetError();

    if ((error != MOTION_MANAGER_ERROR_NONE) &&
        (error != s_previousMotionError))
    {
        Beep_Long();
    }
    s_previousMotionError = error;

4.8 App_Init()：先关闭中断，再从底到顶搭系统

void App_Init(void)
{
    __disable_irq();

    SYSCFG_DL_init();
    Tick_Init();

    LED_Init();
    Beep_Init();
    Key_Init();
    Graydetect_Init();
    Motor_Init();
    Servo_Init();
    Serial1_Init();
    K230Link_Init();
    Odometry_Init();

    DebugDisplay_Init();
    Heading_Init();
    DebugDisplay_ShowHeadingCalibration(Heading_IsReady());
    Heading_Calibrate();

    /* 标定期间全局中断保持关闭，从正式流程的零时刻重新开始计数。 */
    Tick_Init();
    Odometry_Reset();

    BluetoothDebug_Init();

按键边沿公式拆解

输入：本拍 keyMask 与上一拍 s_previousKeyMask。

过程：假设上一拍为 0001（KEY1 已按住），本拍为 0011（KEY2 刚按下）。取反上一拍得到末四位 1110，再与本拍 0011 相与，结果 0010，只留下 KEY2 的新按下事件。

输出：pressedKeys=0011 表示 KEY1、KEY2 都按着；pressedEdges=0010 表示本拍只有 KEY2 是刚按下。

边界条件：上电时 App 先读取一次当前按键作为 previous，避免“上电就按住”被误认成新按下。当前代码未实现软件消抖，机械抖动仍可能造成额外边沿。

设计原因：Mission 常需要“一次按下只启动一次”，不能每 10ms 重复启动；同时调试显示仍需要知道按键是否持续按住，所以保留 level 与 edge 两份信息。

5. 难点函数拆解

Tick_PollCount() 为什么要关中断

输入：无显式参数；隐含输入是 ISR 正在累计的 s_tickCount。

过程：保存 PRIMASK → 关中断 → 复制计数 → 清零计数 → 恢复原 PRIMASK。

输出：返回一个 0～255 的累计节拍快照。

边界条件：积压达到 255 后饱和；100Hz 下表示最多明确记录约 2.55 秒。更久的阻塞仍只返回 255。volatile 只保证实际访存，不能阻止 ISR 插在“读”和“清零”之间，所以仍需临界区。

设计原因：读取并清空必须看成一个不可拆开的整体，同时临界区只有几条指令，不会明显妨碍实时性。

App_Update() 为什么返回 0/1

输入：一块可写的 App_UpdateContext_t 内存。

过程：检查地址 → 取 Tick → 无 Tick 则休眠 → 有 Tick 才刷新状态、事件、通信、运动、提示和显示。

输出：0 表示没有新上下文，1 表示上下文已完整生成。

边界条件：空指针直接返回；累计多拍时并非机械地把整个 App 重跑多次，而是把总拍数或总 dt 传给多数子系统，只有 Beep 明确逐拍补推进。

设计原因：main 可以非常简单地保证 Mission 只消费完整快照；App 是唯一的系统节拍“总分发站”。

6. 放回总体调用链

启动链：复位启动代码 → main() → App_Init() → SysConfig/驱动/状态/通信/运动初始化 → Mission_Init(AccomplishTest_GetMissionGraph()) → Interrupt_Enable()

时间链：32MHz 时钟 → SysTick 每 320000 周期到期 → SysTick_Handler() 累加 → Tick_PollCount() 原子取走 → elapsedTicks → dt

每次有效更新：Heading → Odometry → 按键 level/edge → K230Link → UART1 调试信号 → C0 全局停车 → MotionManager → 错误蜂鸣 → K230 回执 → Telemetry → Beep → DebugDisplay → Mission_Update()

运动主链：main → App → Mission → MotionManager → 具体运动算法 → Motor/Encoder/传感器

顺序不是随便排列的：先更新航向和里程，运动控制才拿到新反馈；先处理 C0，Mission 的不可打断状态也挡不住急停；App 完成所有公共更新后，Mission 才根据同一拍事件决定任务状态是否转换。

7. 常见误解与调试提示

  • “README 说什么就运行什么”：错。当前 main 明确加载 AccomplishTest；README 的 25H 描述已经落后。
  • “无限循环会让 CPU 永远满速空转”：不完全。无 Tick 时 App 执行 __WFI() 等中断，降低无意义忙等。
  • “100Hz 表示 App 每次一定只收到 1 拍”：错。主循环若被较慢操作拖住，ISR 会累计多拍，App 可能收到 2、3……255。
  • “volatile 已经保证安全”：错。读后清零是两个操作，仍必须用关中断临界区防止丢拍。
  • “pressedKeys 与 pressedEdges 一样”：错。前者按住多久就保持多久，后者只在 0→1 的那一拍出现。
  • “关中断就是关掉外设”：错。它只是暂时不让 CPU 响应普通 IRQ；外设配置和挂起状态可能仍在。
  • “Delay 很适合主循环定时”：错。它阻塞业务，且实际时间会叠加函数开销和中断影响。

修改这些参数会牵动什么

  • 修改 100Hz 时，必须同步修改 main.syscfg 的 SysTick period、TICK_HZ、TICK_DT，并复查 K230 重试/超时、蓝牙空闲判帧、遥测频率、OLED 刷新、蜂鸣状态机、里程速度计算和 Mission 中所有“多少拍”的超时。
  • 改变 App_Update() 顺序时，要检查数据依赖：MotionManager 是否仍在状态估计之后，C0 是否仍在 Mission 之前，显示/遥测看到的是更新前还是更新后数据。
  • 在有效更新中加入阻塞串口、长 Delay 或重计算，会增加 elapsedTicks；超过 255 拍后时间信息饱和丢失，控制响应也会明显变差。
  • 提前调用 Interrupt_Enable() 会让 ISR 访问半初始化对象；延后太久则可能丢失本应采集的输入。当前边界是 App 和 Mission 都准备完成之后。

8. 学完自测

1. 当前固件实际加载 25H 还是 AccomplishTest？证据在哪两行？

加载 AccomplishTest。证据是 main.c 第 1 行包含 Accomplish/Test.h，第 11 行把 AccomplishTest_GetMissionGraph() 交给 Mission_Init()。

2. 为什么 Tick 计数器既写 volatile，又要在读取清零时关中断？

volatile 防止编译器缓存读写；关中断则保证“复制并清零”不会被 ISR 插入，从而避免刚增加的一拍被清掉。两者解决的问题不同。

3. elapsedTicks 为 3 时，dt 是多少？为什么不能永远写死 0.01？

dt = 3 × 0.01 = 0.03s。主循环可能因其他工作漏过多个节拍，使用实际累计时间才能让积分、速度和控制更接近真实时间。

4. 上一拍按键位图 0001、本拍 0011，pressedEdges 是多少？

是 0010。KEY1 只是继续按住，KEY2 才是本拍从 0 变为 1。

5. 为什么 main 只在 App_Update 返回非零时调用 Mission_Update？

非零表示 App 已经收到 Tick 并完整更新上下文；返回 0 时没有新的时间片或参数无效，Mission 不应重复消费旧事件和旧时间。
