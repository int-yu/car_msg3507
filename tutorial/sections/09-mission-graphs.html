模块 09：25E、25H 与测试任务图——用静态表给通用 Mission 装上不同题目

一句话职责：三个 Accomplish 模块只描述“有哪些状态、每个状态启动什么动作、什么条件切到哪里”，真正执行状态机的是公共 Mission。

上游输入：App_UpdateContext 的按键按下沿与经过节拍、灰度状态、连续航向、MotionManager 动作状态。

下游输出：静态 Mission_StateDefinition_t 状态表、Mission_Transition_t 转换表和 Mission_GraphDefinition_t 图描述。

依赖：Mission 公共类型与执行器、MotionManager、Heading、Graydetect。

调用者：main.c 选择一张图传给 Mission_Init；Mission 按表调用 Accomplish 的 onEnter、onUpdate 和条件函数。

总体位置：按键/传感器 → AppContext → Mission 引擎 → Accomplish 回调 → MotionManager → 底层运动。

先纠正最重要的理解

Accomplish/25E.c、25H.c、Test.c 不是三套新的状态机引擎。它们是三份“任务数据 + 小回调”：用 Mission.h 的公共结构体声明静态数组。循环调度、错误处理、转换顺序、停止当前运动、进入目标状态都由 Application/Mission/Mission.c 完成。

当前烧进程序的是哪张图？

当前 main.c 第 1 行包含 Accomplish/Test.h，第 11 行调用 Mission_Init(AccomplishTest_GetMissionGraph())。所以现在按 KEY2 执行刹车测试；25E 与 25H 虽然源码完整存在，却没有被 main 选中。要运行正式题目必须明确改 main 的 include 和 GetMissionGraph 调用。

2. 初学者需要先知道

有限状态机把流程分成有限个状态，并规定何时从一个状态转到另一个状态。这里 Mission 是执行器，Accomplish 数组是流程说明书。

回调函数是“把函数地址交给框架，等框架在合适时机调用”。onEnter 进入状态时一次，onUpdate 每个有效系统拍一次，condition 在检查转换时调用。

函数指针保存函数入口地址。Mission_StateDefinition_t 的 onEnter/onUpdate/onExit 字段和 Mission_Transition_t 的 condition 字段都是函数指针。

静态表在编译期确定、程序运行时只读的数组。任务图不动态申请内存，状态和转换关系一目了然。

指定下标初始化例如 [STATE_LINE] = {...}，明确把结构体放到枚举对应槽位；即使枚举顺序调整，也比只靠位置更易检查。

打断转换可在当前动作仍 RUNNING 时生效，例如巡线中发现标志立即切走。普通转换只在动作 FINISHED 后检查。

绝对航向目标表示连续角度坐标中的指定方向。Heading 不限制在 ±180°，所以目标可以是 360°、−270°等连续累计值。

位掩码用整数各位表示多个传感器。0x03 二进制是 00000011，表示灰度 bit0 与 bit1 都要关注。

3. 文件导读

  • 25E.h：只放实车参数和取图接口；25E.c：五状态图、见线确认和 +180°绝对目标循环。
  • 25H.h：左侧标志、前移和转角参数；25H.c：五状态图、左双黑条件和 −90°目标循环。
  • Test.h：刹车测试参数；Test.c：等待、直行、刹车、错误四状态图。

4. 逐行讲解

4.1 Accomplish/25E.h：25E 可调参数

#ifndef ACCOMPLISH_25E_H
#define ACCOMPLISH_25E_H

#include "Application/Mission/Mission.h"

/*
 * 25E 题目流程：
 * 等待 KEY1 按下 -> 最多直行 2000 mm -> 检测到黑线后巡线
 * -> 连续丢线 50 拍后转向绝对目标 -> 重新直行并循环。
 *
 * 本文件只保存需要按题目或实车调整的参数。
 * 距离单位为 mm，速度单位为 mm/s，角度单位为 °。
 */

/* KEY1 在 App 按键位图中对应 bit0；使用按下沿启动，长按不会重复触发。 */
#define ACCOMPLISH_25E_START_KEY_MASK              0x01U

/* 每轮直线寻找黑线的最大距离；走满仍未找到线时停车并回到等待。 */
#define ACCOMPLISH_25E_STRAIGHT_DISTANCE_MM        2000U

/* 直线阶段的巡航速度。 */
#define ACCOMPLISH_25E_STRAIGHT_SPEED_MMPS         300.0f

/* 直线走满最大距离后的终点速度；当前为 0，表示平滑减速至停车。 */
#define ACCOMPLISH_25E_STRAIGHT_END_SPEED_MMPS     0.0f

/* 检测到黑线并进入 MotionLine 后使用的巡线速度。 */
#define ACCOMPLISH_25E_LINE_SPEED_MMPS             200.0f

/* 直线阶段必须连续检测到黑线的节拍数；3 拍在 100 Hz 下约为 30 ms。 */
#define ACCOMPLISH_25E_LINE_DETECT_CONFIRM_TICKS   3U

/*
 * 每次转向时在上一绝对目标上增加的角度。
 * 以 KEY1 启动航向为基准，目标依次增加 180°、360°……
 */
#define ACCOMPLISH_25E_TURN_TARGET_OFFSET_DEG      180.0f

/* Nav 转向时每侧车轮的目标速度。 */
#define ACCOMPLISH_25E_TURN_SPEED_MMPS             80.0f

/*
 * 返回 25E 的只读静态状态图。
 * main.c 把该指针传给 Mission_Init()；调用者不得修改状态表内容。
 */
const Mission_GraphDefinition_t *Accomplish25E_GetMissionGraph(void);

#endif
  1. 第 1～4 行：保护宏并引入 Mission_GraphDefinition_t 类型；头文件不需要知道 Mission 如何执行。
  2. 第 6～13 行：先用自然语言定义流程和单位，防止把 300.0f 误当 PWM 或把角度当弧度。
  3. 第 15～16 行：KEY1 对应 bit0，掩码 0x01。
  4. 第 18～25 行：每轮最多直行 2000 mm，巡航 300 mm/s，末速 0 表示到距离上限时减速停车。
  5. 第 27～31 行：巡线速度 200 mm/s；直线阶段需连续 3 个真实系统拍检测到线，约 30 ms。
  6. 第 33～40 行：每轮绝对目标增加 180°，转向轮速 80 mm/s。这里不是“每次相对转180°”接口，而是代码累加后调用 TurnTo。
  7. 第 42～46 行：唯一公开函数返回 const 图指针；调用者可读但不应改静态表。
  8. 第 48 行：结束保护。

4.2 Accomplish/25E.c：状态编号、运行期记忆与动作入口

#include "Accomplish/25E.h"
#include "Application/Control/MotionManager.h"
#include "Application/State/Heading.h"
#include "Hardware/Sensors/Graydetect.h"
#include <stddef.h>

/*
 * 25E 状态图全部保存在本文件，不向其他模块公开状态编号或回调。
 * Mission 负责执行状态图，MotionManager 负责保证只有一种运动占用双轮。
 *
 * 状态流程：
 * WAITING --KEY1 按下--> STRAIGHT --检测到黑线--> LINE
 * STRAIGHT --走满 2000 mm 未找到线--> WAITING
 * LINE --连续丢线 50 拍--> TURN --到达启动航向 + 180°--> STRAIGHT
 */

/* 根据静态数组自动计算转换数量，避免手工填写后与数组长度不一致。 */
#define ACCOMPLISH_25E_ARRAY_COUNT(array) \
    ((uint8_t)(sizeof(array) / sizeof((array)[0])))

/* 状态编号只在 25E 内部使用，最后的 COUNT 用于定义和校验状态表长度。 */
typedef enum
{
    ACCOMPLISH_25E_STATE_WAITING = 0, /* 等待 KEY1 按下，不驱动车轮。 */
    ACCOMPLISH_25E_STATE_STRAIGHT,    /* 直行并在途中检查灰度黑线。 */
    ACCOMPLISH_25E_STATE_LINE,        /* 使用五路灰度持续巡线。 */
    ACCOMPLISH_25E_STATE_TURN,        /* 指向启动航向加 180° 的绝对目标。 */
    ACCOMPLISH_25E_STATE_ERROR,       /* 运动启动或运行异常后的安全状态。 */
    ACCOMPLISH_25E_STATE_COUNT
} Accomplish25E_StateId_t;

/*
 * 直线阶段的黑线连续检测计数。
 * 进入每一轮直线状态时清零，防止上一轮残留值直接触发巡线。
 */
static uint8_t s_lineDetectedTicks;

/*
 * KEY1 启动本轮任务时的连续航向角。
 * Heading 的角度不做正负 180° 限幅，因此这里保存的也是连续绝对角。
 */
static float s_startYawDeg;
static uint8_t s_startYawCaptured;

/*
 * 每次回到等待状态都清除上一轮任务的航向基准。
 * 下一次按 KEY1 进入直线状态时会重新记录当前航向。
 */
static Mission_CallbackResult_t Accomplish25E_WaitingEnter(void)
{
    s_startYawDeg = 0.0f;
    s_startYawCaptured = 0U;
    return MISSION_CALLBACK_OK;
}

/*
 * 等待状态必须持续返回 RUNNING。
 * 如果 onUpdate 为空或返回 FINISHED，Mission 将不会检查 KEY1 打断转换。
 */
static Mission_ActionStatus_t Accomplish25E_WaitingUpdate(float dt)
{
    (void)dt;
    return MISSION_ACTION_RUNNING;
}

/*
 * 每次进入直线状态只调用一次。
 * 先清除入线确认计数，再启动最大距离 2000 mm 的直线运动。
 * MotionManager 启动失败时返回 ERROR，由 Mission 停车并进入错误状态。
 */
static Mission_CallbackResult_t Accomplish25E_StraightEnter(void)
{
    s_lineDetectedTicks = 0U;

    if (s_startYawCaptured == 0U)
    {
        /* 首次直行开始前记录 KEY1 启动时的航向，后续循环不再覆盖。 */
        s_startYawDeg = Heading_GetYaw();
        s_startYawCaptured = 1U;
    }

    return (MotionManager_StartForward(
                ACCOMPLISH_25E_STRAIGHT_DISTANCE_MM,
                ACCOMPLISH_25E_STRAIGHT_SPEED_MMPS,
                ACCOMPLISH_25E_STRAIGHT_END_SPEED_MMPS) ==
            MOTION_MANAGER_RESULT_OK) ?
        MISSION_CALLBACK_OK : MISSION_CALLBACK_ERROR;
}
  1. 第 1～5 行：25E 依赖公共 Mission 类型、运动管理、航向、灰度和 NULL。
  2. 第 7～15 行：注释再次强调职责分层并给出状态边。Accomplish 不在 onUpdate 里直接驱动轮子。
  3. 第 17～19 行：数组计数宏用“整个数组字节数÷单元素字节数”。它只能接真实数组；若传指针，sizeof 得到指针大小，结果错误。
  4. 第 21～30 行：枚举把五个状态编号固定为 0～4，COUNT 自动成为 5，用来定义数组长度和 graph.stateCount。
  5. 第 32～36 行：s_lineDetectedTicks 是跨 Update 保留的连续见线计数，static 让它只在本文件可见且整个运行期存在。
  6. 第 38～43 行：s_startYawDeg 与 captured 保存一次任务的航向基准。注意后面会在原变量上 +=180，所以转向后它实际保存“最近绝对目标”。
  7. 第 45～54 行：每次进入 WAITING 清除旧基准并返回 OK；下次启动会重新以当前车头方向为起点。
  8. 第 56～64 行：等待状态丢弃 dt 并永远返回 RUNNING。Mission 只有在动作 RUNNING 且 interruptible=1 时检查打断转换，因此不能把它写成 FINISHED。
  9. 第 66～73 行：StraightEnter 每次进入只执行一次，先清见线计数。
  10. 第 75～80 行：只有第一轮直线捕获 Heading_GetYaw；后续 TURN→STRAIGHT 循环不会覆盖基准。
  11. 第 82～88 行：启动 2000 mm 直线并用三目运算把 MotionManager_Result 映射成 Mission_CallbackResult；失败会让 Mission 进入 ERROR。
/*
 * 检测到黑线后进入本状态，只调用一次 MotionManager_StartLine()。
 * MotionLine 内部完成五路权重差速，并在连续全白 50 拍后返回 FINISHED。
 */
static Mission_CallbackResult_t Accomplish25E_LineEnter(void)
{
    return (MotionManager_StartLine(
                ACCOMPLISH_25E_LINE_SPEED_MMPS) ==
            MOTION_MANAGER_RESULT_OK) ?
        MISSION_CALLBACK_OK : MISSION_CALLBACK_ERROR;
}

/*
 * 巡线完成后进入本状态，在上一绝对目标上增加 180°。
 * 使用 TurnTo 依次指向启动航向 +180°、+360°、+540°……
 */
static Mission_CallbackResult_t Accomplish25E_TurnEnter(void)
{
    if (s_startYawCaptured == 0U)
    {
        /* 未保存启动航向说明状态流程异常，禁止使用错误基准启动转向。 */
        return MISSION_CALLBACK_ERROR;
    }

    return (MotionManager_TurnTo(
                s_startYawDeg += ACCOMPLISH_25E_TURN_TARGET_OFFSET_DEG,
                ACCOMPLISH_25E_TURN_SPEED_MMPS) ==
            MOTION_MANAGER_RESULT_OK) ?
        MISSION_CALLBACK_OK : MISSION_CALLBACK_ERROR;
}

/*
 * 直线、巡线和转向共用的状态更新回调。
 * 底层运动已经由 App_Update() 调用 MotionManager_Update(dt) 更新，
 * 此处只查询动作是运行、完成还是错误，不能再次更新底层控制器。
 */
static Mission_ActionStatus_t Accomplish25E_MotionUpdate(float dt)
{
    (void)dt;
    return Mission_GetMotionActionStatus();
}

/*
 * 等待状态的启动条件。
 * App 只在 KEY1 从松开变为按下的那一拍设置 pressedEdges bit0。
 * 使用按下沿而不是 pressedKeys，可以避免长按 KEY1 时反复启动流程。
 */
static uint8_t Accomplish25E_ConditionStart(
    const Mission_Runtime_t *runtime,
    const App_UpdateContext_t *updateContext)
{
    (void)runtime;
    if (updateContext == NULL)
    {
        return 0U;
    }

    return ((updateContext->pressedEdges &
             ACCOMPLISH_25E_START_KEY_MASK) != 0U) ? 1U : 0U;
}
  1. 第 90～100 行：LineEnter 只启动一次 MotionLine；连续全白 50 拍是 MotionLine 公共配置决定的“正常完成”，不是 25E 自己计数。
  2. 第 102～112 行：TurnEnter 先确认基准存在，防止异常状态路径用 0°乱转。
  3. 第 114～119 行：表达式 s_startYawDeg +=180 先修改变量再把新值传给 TurnTo。若启动航向 12°，目标依次为 192°、372°、552°。
  4. 第 121～130 行：公共 MotionUpdate 不重复调用 MotionManager_Update；App 已先更新控制器，这里只把 error/finished/busy 映射为 Mission 动作状态。
  5. 第 132～141 行：ConditionStart 的两个参数符合公共条件回调签名；runtime 本条件不用，以 (void) 避免警告。
  6. 第 142～149 行：空上下文不触发；pressedEdges & 0x01 非零才返回 1，因此长按 KEY1 只在首拍启动。
/*
 * 直线阶段的入线条件：五路灰度任意一路为 1 都表示检测到黑线。
 * 为避免单拍毛刺误触发，必须连续达到头文件设定的确认节拍数。
 * 使用 updateContext->elapsedTicks 累加真实经过的系统拍数，主循环偶尔延迟时
 * 仍能保持正确的确认时间，而不是固定按一次函数调用等于 10 ms 计算。
 */
static uint8_t Accomplish25E_ConditionLineDetected(
    const Mission_Runtime_t *runtime,
    const App_UpdateContext_t *updateContext)
{
    uint16_t detectedTicks;

    (void)runtime;
    if (updateContext == NULL)
    {
        return 0U;
    }

    if (Graydetect_OnLine(GRAY_SIDE_ALL) == 0U)
    {
        /* 任意一拍恢复全白都重新开始连续检测。 */
        s_lineDetectedTicks = 0U;
        return 0U;
    }

    detectedTicks = (uint16_t)s_lineDetectedTicks +
                    updateContext->elapsedTicks;
    if (detectedTicks >= ACCOMPLISH_25E_LINE_DETECT_CONFIRM_TICKS)
    {
        /* 达到阈值后进行饱和，避免计数继续累加溢出。 */
        s_lineDetectedTicks =
            ACCOMPLISH_25E_LINE_DETECT_CONFIRM_TICKS;
        return 1U;
    }

    s_lineDetectedTicks = (uint8_t)detectedTicks;
    return 0U;
}
  1. 第 151～159 行：见线条件要求五路中任意一路为黑，并用 App 的真实 elapsedTicks 做连续确认。
  2. 第 161～167 行：用 uint16_t 临时和防止 uint8_t 加法先回绕；空 context 返回假。
  3. 第 169～174 行：Graydetect_OnLine(GRAY_SIDE_ALL)=0 表示全白，立即把连续计数清零，所以条件是“连续”而非“累计总共”。
  4. 第 176～178 行：旧计数加真实经过拍数；主循环若一次补 2 拍，就加 2 而不是只加 1。
  5. 第 178～184 行：达到 3 拍后把状态饱和在 3 并返回真，防止后续继续增长溢出。
  6. 第 186～188 行：未达阈值时保存新计数并继续留在 STRAIGHT。

难点拆解：同一拍既到 2000 mm 又确认黑线

输入：MotionManager 本拍报告 FINISHED，灰度连续确认也达到 3 拍。

4.4 Accomplish/25H.c：基准航向、动作入口与标志条件

#include "Accomplish/25H.h"
#include "Application/Control/MotionManager.h"
#include "Application/State/Heading.h"
#include "Hardware/Sensors/Graydetect.h"
#include <stddef.h>

/*
 * 25H 只描述题目流程，不直接更新底层运动控制器。
 * App 每拍更新 MotionManager，Mission 再根据本状态图决定是否切换动作。
 *
 * 状态流程：
 * WAITING --KEY1--> LINE --左侧两路同时为 1--> FORWARD
 * FORWARD --完成 150 mm--> TURN --到达绝对目标--> LINE
 * LINE --连续丢线并停止--> WAITING
 */

#define ACCOMPLISH_25H_ARRAY_COUNT(array) \
    ((uint8_t)(sizeof(array) / sizeof((array)[0])))

typedef enum
{
    ACCOMPLISH_25H_STATE_WAITING = 0, /* 等待 KEY1，不驱动车轮。 */
    ACCOMPLISH_25H_STATE_LINE,        /* 使用五路灰度巡线。 */
    ACCOMPLISH_25H_STATE_FORWARD,     /* 离开标志线并直行 150 mm。 */
    ACCOMPLISH_25H_STATE_TURN,        /* 转到下一绝对左转目标。 */
    ACCOMPLISH_25H_STATE_ERROR,       /* 运动异常后的安全停车状态。 */
    ACCOMPLISH_25H_STATE_COUNT
} Accomplish25H_StateId_t;

/*
 * s_nextTurnTargetDeg 保存下一次 TurnTo() 使用的连续绝对角。
 * KEY1 启动时以当前航向作为 0°基准；每进入一次 TURN 就减去 90°。
 */
static float s_nextTurnTargetDeg;
static uint8_t s_headingReferenceCaptured;

/* 返回等待状态时清除上一轮任务的航向基准。 */
static Mission_CallbackResult_t Accomplish25H_WaitingEnter(void)
{
    s_nextTurnTargetDeg = 0.0f;
    s_headingReferenceCaptured = 0U;
    return MISSION_CALLBACK_OK;
}

/* 等待状态保持 RUNNING，才能检查 KEY1 的打断转换。 */
static Mission_ActionStatus_t Accomplish25H_WaitingUpdate(float dt)
{
    (void)dt;
    return MISSION_ACTION_RUNNING;
}

/*
 * 首次进入巡线时记录 KEY1 启动方向；后续循环不再覆盖该基准。
 * MotionLine 连续丢线达到公共配置值后会正常结束并回到等待。
 */
static Mission_CallbackResult_t Accomplish25H_LineEnter(void)
{
    if (s_headingReferenceCaptured == 0U)
    {
        if (Heading_IsReady() == 0U)
        {
            /* 本题后续必须使用绝对角转向，MPU 离线时禁止启动车辆。 */
            return MISSION_CALLBACK_ERROR;
        }
        s_nextTurnTargetDeg = Heading_GetYaw();
        s_headingReferenceCaptured = 1U;
    }

    return (MotionManager_StartLine(
                ACCOMPLISH_25H_LINE_SPEED_MMPS) ==
            MOTION_MANAGER_RESULT_OK) ?
        MISSION_CALLBACK_OK : MISSION_CALLBACK_ERROR;
}

/* 检测到左侧标志后，使用航向闭环向前直行 150 mm 并平滑停车。 */
static Mission_CallbackResult_t Accomplish25H_ForwardEnter(void)
{
    return (MotionManager_StartForward(
                ACCOMPLISH_25H_FORWARD_DISTANCE_MM,
                ACCOMPLISH_25H_FORWARD_SPEED_MMPS,
                ACCOMPLISH_25H_FORWARD_END_SPEED_MMPS) ==
            MOTION_MANAGER_RESULT_OK) ?
        MISSION_CALLBACK_OK : MISSION_CALLBACK_ERROR;
}

/*
 * 每次进入 TURN 都先生成新的绝对目标，再调用 TurnTo()。
 * 例如启动航向为 12°，目标依次为 -78°、-168°、-258°……
 */
static Mission_CallbackResult_t Accomplish25H_TurnEnter(void)
{
    if (s_headingReferenceCaptured == 0U)
    {
        return MISSION_CALLBACK_ERROR;
    }

    s_nextTurnTargetDeg += ACCOMPLISH_25H_TURN_STEP_DEG;
    return (MotionManager_TurnTo(
                s_nextTurnTargetDeg,
                ACCOMPLISH_25H_TURN_SPEED_MMPS) ==
            MOTION_MANAGER_RESULT_OK) ?
        MISSION_CALLBACK_OK : MISSION_CALLBACK_ERROR;
}
  1. 第 1～15 行：依赖与职责和 25E 相似，但状态流程从 LINE 开始，没有“先直行找线”。
  2. 第 17～28 行：数组计数宏和五状态枚举；COUNT 同样用于表长。
  3. 第 30～35 行：s_nextTurnTargetDeg 初始保存启动航向，之后每轮加 −90；captured 防止循环重采样。
  4. 第 37～43 行：回 WAITING 清除上一轮基准。
  5. 第 45～50 行：等待持续 RUNNING 以接收 KEY1 打断。
  6. 第 52～64 行：首次 LineEnter 先要求 Heading ready；25H 后续必做绝对转向，所以 MPU 不可用时宁可进入错误状态也不启动车。
  7. 第 65～73 行：保存当前连续 yaw 后启动 200 mm/s 巡线；后续 TURN→LINE 不再覆盖目标基准。

难点拆解：为什么刹车是独立状态

输入：FORWARD 已按速度曲线到达 500 mm并以目标末速 0 正常结束。

过程：Mission 正常转入 BRAKE；BrakeEnter 请求 MotionManager 的刹车模式，公共层按“释放一段时间→主动短刹→最终释放”推进。

输出：刹车动作 FINISHED 后回 WAITING。

边界：任何阶段出错都转 ERROR；C0 仍能从外部全局复位并释放电机。

设计原因：把定距减速和主动刹车分成两个可观测动作，便于比较滑行距离，也保持所有电机所有权在 MotionManager。

5. 难点：这些静态表怎样被公共 Mission 执行

Mission.h 定义统一插槽：一个状态包含 onEnter、onUpdate、onExit、transitions、transitionCount、interruptible；一条转换包含 condition、targetState、type；整张图只包含状态数组指针、状态数、起始状态、错误状态。

  1. Mission_Init 先校验图非空、状态数非零、起始/错误编号合法、每个转换目标合法，再进入 startState。
  2. 进入状态时先把 actionStatus 设为 RUNNING（有 onUpdate）或 FINISHED（无 onUpdate），然后调用一次 onEnter。onEnter 返回 ERROR 就进入错误流程。
  3. App_Update 已先执行 MotionManager_Update(dt)，随后 main 才调用 Mission_Update。因此 Accomplish 的 MotionUpdate 只查询，不能重复推进底层控制。
  4. 动作仍 RUNNING 时，只有 interruptible=1 才扫描 MISSION_TRANSITION_INTERRUPT。
  5. 动作 FINISHED 时扫描 MISSION_TRANSITION_NORMAL；condition=NULL 表示无条件。
  6. Mission_TryTransition 从数组下标 0 向后找，第一条“类型匹配且条件真”的转换获胜；随后调用 onExit、MotionManager_Stop，再进入 targetState。
  7. 动作或 MotionManager 报错时进入 graph.errorState。C0 全局停车在错误状态判断之前处理，可回到 startState。

难点拆解：ARRAY_COUNT 为什么比手写数字安全

输入：一个含 3 个 Mission_Transition_t 元素的真实数组。

过程：sizeof(array) 得到整个数组字节数，sizeof(array[0]) 得到单元素字节数，两者相除得到 3，再转成 uint8_t。

输出：transitionCount 自动随增删元素变化，避免数组已有 3 条但手写 count=2。

边界：函数参数中的 array 会退化为指针；把指针传给宏只会得到“指针大小÷元素大小”。本项目只在静态数组定义所在文件使用。

设计原因：让数据表自己携带可由编译器计算的长度，减少维护重复信息。

难点拆解：25E 与 25H 的绝对目标如何累积

输入：启动时 Heading_GetYaw()=12°。

过程：25E 首次直行保存 12，进入 TURN 时每次 +=180，依次传 192、372、552；25H 首次巡线保存 12，每次 +=(-90)，依次传 −78、−168、−258。

输出：两者都调用 TurnTo 的连续绝对目标，避免每轮相对误差简单累积到目标定义中。

边界：回 WAITING 会清 captured，下一次按键重新以当前航向为新基准；Heading 未做 ±180°包裹，Nav 必须理解连续角。

设计原因：题目循环目标相对于启动方向稳定，车体中途小偏差不会把下一目标改成“当前误差再转固定角”。

6. 放回总体调用链

实体 KEY1/KEY2 → Key_GetPressedMask → App_Update pressedEdges → App_UpdateContext → Mission condition 回调
main 选择 GetMissionGraph → Mission_Init 校验静态图 → 进入 WAITING
Mission onEnter → Accomplish StartForward/StartLine/TurnTo/StartBrake → MotionManager 独占双轮
App MotionManager_Update(dt) → Mission onUpdate → Mission_GetMotionActionStatus → RUNNING/FINISHED/ERROR → 转换表

main.c 第 10 行 App_Init 建好硬件与 MotionManager，第 11 行只选择一张图，第 12 行开中断。主循环第 16～18 行每有有效 App 节拍就把同一份 updateContext 交给 Mission。更换题目通常不改 Mission.c，只改 main 包含的 Accomplish 头文件和传入的 getter。

所有状态的 onExit 当前都是 NULL，因为公共 Mission 在任何成功转换的第 172～174 行会先处理退出，再统一 MotionManager_Stop，确保旧动作释放双轮后才调用新状态 onEnter。Accomplish 不需要在每张图重复 Stop。

MotionManager 同一时刻只允许 STRAIGHT、LINE、TURN、BRAKE 等一种模式。任务图表达顺序，MotionManager 管动作所有权，MotionStraight/MotionLine/Nav 管具体闭环，Motor 才管最终引脚；层次不能颠倒。

7. 常见误解与修改联动

“interruptible=0 就什么都停不了。”它只禁止该状态的普通 INTERRUPT 转换。C0 是 Mission_Update 最前面的全局复位，MotionManager 错误也会强制进 ERROR。

“condition=NULL 表示转换无效。”相反，它表示无条件。NORMAL 数组中的 NULL 常作为动作完成后的唯一去向或最后兜底，放置顺序非常重要。

“onUpdate 应该再次调用 MotionManager_Update。”不能。App 已经更新一次；重复更新会让同一个真实 10 ms 被计算两次，距离、PID、转向和刹车计时都会变快。

“25H 的 0x03 表示灰度状态必须恰好等于3。”条件先与 0x03，再比较 0x03，因此只要求低两位都为1；例如 0x07、0x1B 同样触发。

“修改了 25E 参数，车就会跑25E。”当前 main 仍选 Test。参数文件存在不代表图已加载，必须检查 main.c 的 include 和 Mission_Init 实参。

修改任务图时必须一起检查

  • 增加状态：在 enum 的 COUNT 之前加入编号，在 stateDefinitions 增加指定初始化，并确认 graph.stateCount 自动随 COUNT 变化。
  • 增加转换：用 ARRAY_COUNT 自动更新数量；检查它应在 RUNNING 时打断还是 FINISHED 后正常发生，并检查数组优先级。
  • 改变启动键：同步掩码与人机说明；KEY1=0x01、KEY2=0x02，条件读取的是 pressedEdges 而非 pressedKeys。
  • 改变直线末速：Test 的末速必须允许 MotionStraight 按预期完成；25H 转向前需要稳定零速。
  • 改变绝对转角：同时确认 Heading 连续角方向、Nav 的正负方向约定、机械场地转向方向和目标累积变量。
  • 改变灰度标志：确认 Graydetect 位序、是否需要连续多拍确认，以及动作完成同拍是否要补 NORMAL 条件转换。
  • 切换正式任务：main.c 只能选择一个 GetMissionGraph；构建前在启动提示、按键和路线说明中同步当前图。

8. 学完自测

为什么说 25E.c 不是第二套状态机引擎？

它只定义 Mission 公共结构体数组和回调；状态进入、更新、错误、转换扫描与停止动作都由 Mission.c 执行。

WAITING 的 onUpdate 为什么必须返回 RUNNING？

Mission 只在 RUNNING 且 interruptible=1 时检查 INTERRUPT 转换；若 WAITING 直接 FINISHED，KEY 按下沿转换不会按设计被检查。

25E 在同拍到达2000mm并确认黑线时去哪里？

去 LINE。动作已完成时检查 NORMAL，见线 NORMAL 条件排在 NULL→WAITING 之前。

25H 灰度为0x07时左标志条件成立吗？

成立。0x07 & 0x03=0x03，低两位同时为1，其他位不影响。

启动航向12°时，25H前两次TurnTo目标是什么？

−78°、−168°；每次在连续绝对目标上加−90°。

Test 为什么不直接调用 Motor_Brake？

电机所有权必须由 MotionManager 统一调度；独立 BRAKE 状态请求 StartBrake，公共层负责刹车阶段和最终释放。

interruptible=0 能否阻止C0停车？

不能。C0 是 Mission 的全局复位路径，在普通状态转换和错误状态判断之前处理。

当前按哪个键运行什么任务？

当前 main 加载 AccomplishTest 图，按 KEY2 运行500mm直行后主动刹车测试；KEY1不会启动25E或25H。
