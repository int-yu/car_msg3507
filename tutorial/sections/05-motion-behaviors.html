05　直行、巡线、转向与运动管理：把任务意图变成左右轮目标

先看它在整车中的位置

一句话职责
把 Mission 的“走 500 mm、沿黑线走、转到 90°、短暂刹车”等动作请求，转换为每个控制周期的左右轮速度目标，并保证同一时刻只有一种行为控制电机。
上游输入
Mission/蓝牙命令给出的距离、速度、终点速度和角度；Heading、Odometry、Graydetect 提供的航向、路程、轮速与灰度状态；App 提供的秒单位 dt。
下游输出
提交给 MotionWheel 的左右轮目标速度与附加 PWM 修正；同时向 Mission 返回 busy、finished、error 和当前模式。
依赖模块
MotionWheel、PID、Heading、Odometry、Graydetect、底层 Motor。
主要调用者
题目状态机与通用 Mission、蓝牙调试命令；App_Update 每拍调用 MotionManager_Update(dt)。
总体位置
Mission → MotionManager → Straight / Line / Nav → MotionWheel → Motor。状态数据则沿 Heading/Odometry/Graydetect → 行为模块 反向反馈。

初学者需要先知道

运行时上下文：一个结构体集中保存动作进行到哪里、目标是什么、上次算到什么。像一张正在执行订单的工作单，函数每拍都读写同一张单。

状态机：用 IDLE、RUNNING、COMPLETED、ERROR 等离散状态限制代码路径。状态不是文字说明，而是决定本拍允许执行什么的开关。

参数校验：启动或初始化前检查 NaN、无穷大、负时间、上下限矛盾等。控制器收到非法浮点数时，结果可能不是简单的“跑慢一点”，而是 PWM 完全不可预测。

速度斜坡 / 速度规划：目标速度不瞬间跳变，而是按加速度每拍逐渐接近。这样能减少打滑、电流冲击和姿态摆动。

限幅：把数值限制在最小值和最大值之间。请求 1500 mm/s 而上限是 1000 时，输出 1000。

PD 控制器：P 根据当前误差回正，D 根据误差变化趋势抑制摆动；本模块航向 PID 的 I 为 0，因此实际是 PD。

前馈与反馈：行为层先给出想要的轮速，MotionWheel 再用编码器反馈修正 PWM；Straight 还额外给左右相反的航向 trim。

绝对/相对航向：TurnTo 的目标是连续航向坐标中的一个数；TurnBy 是“当前航向 + 增量”。两者都不自动选择 ±180° 的最短路。

稳定确认：误差进入容差一次不算完成，必须轮速已经降到零且连续多拍仍在容差内。

互斥所有权：Straight、Line、Nav 都能写双轮目标，但管理器一次只更新一个。否则三个控制器会在同一拍抢电机。

忙碌 / 已到分段终点：它们不一定互为反义词。Straight 的 CONTINUING 表示分段距离完成、可让 Mission 转移，但仍持续输出终点轮速，所以 busy 与 finished 同时为 1。

文件导读

文件角色核心输入与输出
MotionStraight.h/.c定距直行行为读左右里程与航向；输出同向轮速、航向差速 trim、剩余距离和完成状态。
MotionLine.h/.c灰度巡线行为读五路灰度位图；输出按权重增减后的左右轮速，并把连续全白解释为正常结束。
Nav.h/.c原地转向行为读连续航向；输出大小相同、方向相反的左右轮速，直到角度稳定。
MotionManager.h/.c统一调度与状态适配接收 Mission 动作请求，只更新当前模式，统一映射启动错误，并管理独立的释放/刹车阶段。

三个行为都调用同一个 MotionWheel，因为轮速 PID、前馈、PWM 限幅、里程反馈和电机方向规则必须只有一份实现。行为层回答“左右轮应该多快”，轮速层回答“为了达到这个速度该给多少 PWM”。

static void MotionStraight_PrepareProfile(
    float distanceMM, float speedMMps, float endSpeedMMps)
{
    s_context.startDistanceLMM = Odometry_GetDistanceLMM();
    s_context.startDistanceRMM = Odometry_GetDistanceRMM();
    s_context.totalDistanceMM = fabsf(distanceMM);
    s_context.remainingDistanceMM = distanceMM;
    s_context.direction = (distanceMM >= 0.0f) ? 1.0f : -1.0f;

    s_context.cruiseSpeedMMps = MotionStraight_Clamp(
        speedMMps, 0.0f, MOTION_STRAIGHT_MAX_SPEED_MMPS);
    s_context.endSpeedMMps = MotionStraight_Clamp(
        endSpeedMMps, 0.0f, s_context.cruiseSpeedMMps);

    /*
     * 容差是允许的终点误差，因此速度曲线在 total - tolerance 处达到
     * endSpeed。这样在允许误差内已经是目标速度，而不会到点后突然降速。
     */
    s_context.profileEndDistanceMM = s_context.totalDistanceMM -
        MOTION_STRAIGHT_DISTANCE_TOLERANCE_MM;
    if (s_context.profileEndDistanceMM < 0.0f)
    {
        s_context.profileEndDistanceMM = 0.0f;
    }

    s_context.decelerationStartDistanceMM =
        s_context.totalDistanceMM *
        MOTION_STRAIGHT_DECELERATION_START_RATIO;
    if (s_context.decelerationStartDistanceMM >
        s_context.profileEndDistanceMM)
    {
        /* 极短距离没有完整的后 1/6，直接在速度规划终点前开始减速。 */
        s_context.decelerationStartDistanceMM =
            s_context.profileEndDistanceMM;
    }

    s_context.effectiveDecelerationMMps2 = 0.0f;
    s_context.decelerationActive = 0U;
    s_context.targetReached = 0U;
    s_context.zeroSpeedHoldElapsedSeconds = 0.0f;
}

static float MotionStraight_GetDirectedTravelledDistance(void)
{
    float distanceLMM = Odometry_GetDistanceLMM() -
                        s_context.startDistanceLMM;
    float distanceRMM = Odometry_GetDistanceRMM() -
                        s_context.startDistanceRMM;
    float averageDistanceMM = (distanceLMM + distanceRMM) * 0.5f;

    return s_context.direction * averageDistanceMM;
}

/* 在固定减速点记录当前速度，并计算剩余距离内所需的恒定减速度。 */
static void MotionStraight_StartDecelerationIfNeeded(
    float travelledDistanceMM)
{
    float currentSpeedMMps;
    float profileRemainingDistanceMM;

    if ((s_context.decelerationActive != 0U) ||
        (travelledDistanceMM < s_context.decelerationStartDistanceMM))
    {
        return;
    }

    s_context.decelerationActive = 1U;
    currentSpeedMMps = fabsf(s_context.profileSpeedMMps);
    profileRemainingDistanceMM = s_context.profileEndDistanceMM -
        travelledDistanceMM;
    if ((currentSpeedMMps > s_context.endSpeedMMps) &&
        (profileRemainingDistanceMM > 0.001f))
    {
        /* v_end^2 = v_start^2 - 2 * a * s。a 在本次减速段内保持不变。 */
        s_context.effectiveDecelerationMMps2 =
            (currentSpeedMMps * currentSpeedMMps -
             s_context.endSpeedMMps * s_context.endSpeedMMps) /
            (2.0f * profileRemainingDistanceMM);
    }
}

/* 距离规划层：返回本周期应跟踪的正速度大小。 */
static float MotionStraight_CalculateTargetSpeedMagnitude(void)
{
    float travelledDistanceMM;
    float remainingDistanceMM;
    float profileRemainingDistanceMM;
    float brakingSpeedMMps;

    if (s_context.state == MOTION_STRAIGHT_STATE_CONTINUING)
    {
        s_context.remainingDistanceMM = 0.0f;
        return s_context.endSpeedMMps;
    }

    travelledDistanceMM = MotionStraight_GetDirectedTravelledDistance();
    remainingDistanceMM = s_context.totalDistanceMM -
                          travelledDistanceMM;
    s_context.remainingDistanceMM =
        s_context.direction * remainingDistanceMM;

    MotionStraight_StartDecelerationIfNeeded(travelledDistanceMM);

    profileRemainingDistanceMM = s_context.profileEndDistanceMM -
        travelledDistanceMM;
    if (profileRemainingDistanceMM <= 0.0f)
    {
        /* 到达距离容差范围时，速度已按规划下降至结束速度。 */
        s_context.targetReached = 1U;
        return s_context.endSpeedMMps;
    }
    if (s_context.decelerationActive == 0U)
    {
        return s_context.cruiseSpeedMMps;
    }

    brakingSpeedMMps = sqrtf(
        s_context.endSpeedMMps * s_context.endSpeedMMps +
        2.0f * s_context.effectiveDecelerationMMps2 *
        profileRemainingDistanceMM);
    return MotionStraight_Clamp(
        brakingSpeedMMps,
        s_context.endSpeedMMps,
        s_context.cruiseSpeedMMps);
}

{

    float targetSpeedMMps;

    if ((s_context.state != MOTION_STRAIGHT_STATE_RUNNING) &&
        (s_context.state != MOTION_STRAIGHT_STATE_CONTINUING))
    {
        return;
    }
    if ((!isfinite(dt)) || (dt <= 0.0f))
    {
        MotionStraight_SetError(
            MOTION_STRAIGHT_ERROR_UPDATE_PERIOD_INVALID);
        return;
    }
    if (Heading_IsReady() == 0U)
    {
        MotionStraight_SetError(MOTION_STRAIGHT_ERROR_HEADING_OFFLINE);
        return;
    }
    if (Odometry_CountsPerMM <= 0.001f)
    {
        MotionStraight_SetError(MOTION_STRAIGHT_ERROR_ODOMETRY_INVALID);
        return;
    }

    targetSpeedMagnitudeMMps =
        MotionStraight_CalculateTargetSpeedMagnitude();
    targetSpeedMMps = s_context.direction * targetSpeedMagnitudeMMps;
    MotionStraight_UpdateProfileSpeed(targetSpeedMMps, dt);

    if (MotionStraight_ApplyWheelCommand(dt) != MOTION_WHEEL_RESULT_OK)
    {
        MotionStraight_SetError(MOTION_STRAIGHT_ERROR_WHEEL);
        return;
    }
    if (MotionStraight_FinishProfileIfReady(targetSpeedMMps, dt) != 0U)
    {
        return;
    }
}

/*
 * 位图从左到右为 bit0~bit4，灰度返回 1 表示检测到黑线。
 * 最外侧权重为正负 6，内侧权重为正负 3，中心权重为 0。
 */
static int8_t MotionLine_GetWeight(uint8_t grayState)
{
    int16_t weight = 0;

    if ((grayState & 0x01U) != 0U)
    {
        weight -= MOTION_LINE_OUTER_WEIGHT;
    }
    if ((grayState & 0x02U) != 0U)
    {
        weight -= MOTION_LINE_INNER_WEIGHT;
    }
    if ((grayState & 0x08U) != 0U)
    {
        weight += MOTION_LINE_INNER_WEIGHT;
    }
    if ((grayState & 0x10U) != 0U)
    {
        weight += MOTION_LINE_OUTER_WEIGHT;
    }

    /* 多个同侧探头同时压线时，不允许超过最外侧的修正力度。 */
    if (weight > MOTION_LINE_OUTER_WEIGHT)
    {
        weight = MOTION_LINE_OUTER_WEIGHT;
    }
    else if (weight < -MOTION_LINE_OUTER_WEIGHT)
    {
        weight = -MOTION_LINE_OUTER_WEIGHT;
    }

    return (int8_t)weight;
}

static uint8_t MotionLine_CalculateTargetSpeeds(
    float *leftSpeedMMps, float *rightSpeedMMps, float dt)
{
    uint8_t grayState = Graydetect_GetState();
    float weight;
    float speedAdjustMMps;

    if (grayState == 0U)
    {
        if (s_context.lostTicks < MOTION_LINE_LOST_CONFIRM_TICKS)
        {
            s_context.lostTicks++;
        }
        if (s_context.lostTicks >= MOTION_LINE_LOST_CONFIRM_TICKS)
        {
            return 0U;
        }

        /* 短暂丢线时保持上一拍的左右轮目标速度。 */
        *leftSpeedMMps = s_context.lastLeftSpeedMMps;
        *rightSpeedMMps = s_context.lastRightSpeedMMps;
        return 1U;
    }

    s_context.lostTicks = 0U;
    weight = (float)MotionLine_GetWeight(grayState);
    s_context.lineError = weight;

    /* 权重达到正负 6 时，速度增减比例等于最大调整比例；
     * 微分项对权重跳变（压线切换瞬间）施加一次性阻尼，默认 0 不生效。 */
    speedAdjustMMps = s_context.cruiseSpeedMMps *
                      MotionLine_TuneMaxAdjustRatio *
                      (weight / (float)MOTION_LINE_OUTER_WEIGHT);
    speedAdjustMMps += MotionLine_TuneWeightKd *
                       ((weight - s_context.previousWeight) / dt);
    s_context.previousWeight = weight;

    /* 阻尼过强时不允许反向超过巡航速度，防止单轮猛烈倒转。 */
    if (speedAdjustMMps > s_context.cruiseSpeedMMps)
    {
        speedAdjustMMps = s_context.cruiseSpeedMMps;
    }
    else if (speedAdjustMMps < -s_context.cruiseSpeedMMps)
    {
        speedAdjustMMps = -s_context.cruiseSpeedMMps;
    }

    /* 左侧压线：左轮减速、右轮加速；右侧压线时相反。 */
    *leftSpeedMMps = s_context.cruiseSpeedMMps + speedAdjustMMps;
    *rightSpeedMMps = s_context.cruiseSpeedMMps - speedAdjustMMps;
    s_context.lastLeftSpeedMMps = *leftSpeedMMps;
    s_context.lastRightSpeedMMps = *rightSpeedMMps;
    return 1U;
}

/* 根据剩余角度生成带低速区的有符号转向轮速。 */
static float Nav_CalculateTargetTurnSpeed(void)
{
    float absoluteErrorDeg = fabsf(s_context.angleErrorDeg);
    float minimumSpeedMMps;
    float slowdownRatio;
    float speedMagnitudeMMps;
    float direction;

    if (absoluteErrorDeg <= Nav_TuneAngleToleranceDeg)
    {
        return 0.0f;
    }

    minimumSpeedMMps = Nav_Clamp(
        Nav_TuneMinTurnSpeedMMps,
        0.0f, s_context.requestedSpeedMMps);
    slowdownRatio = Nav_Clamp(
        absoluteErrorDeg / Nav_TuneSlowdownAngleDeg,
        0.0f, 1.0f);
    speedMagnitudeMMps = minimumSpeedMMps +
        (s_context.requestedSpeedMMps - minimumSpeedMMps) *
        slowdownRatio;

    direction = (s_context.angleErrorDeg >= 0.0f) ? 1.0f : -1.0f;
    return direction * (float)NAV_ROTATION_COMMAND_SIGN *
           speedMagnitudeMMps;
}

static void Nav_UpdateSpeedProfile(float targetTurnSpeedMMps, float dt)
{

        ((targetTurnSpeedMMps * s_context.profileTurnSpeedMMps) < 0.0f) ?
            1U : 0U;

        (fabsf(targetTurnSpeedMMps) <
         fabsf(s_context.profileTurnSpeedMMps)))
    {
        speedStepMMps = NAV_DECELERATION_MMPS2 * dt;
    }
    else
    {
        speedStepMMps = NAV_ACCELERATION_MMPS2 * dt;
    }
    s_context.profileTurnSpeedMMps = Nav_Approach(
        s_context.profileTurnSpeedMMps,
        targetTurnSpeedMMps,
        speedStepMMps);
}

static void MotionManager_UpdateBrake(float dt)
{

    float brakeEndSeconds;

    if ((!isfinite(dt)) || (dt <= 0.0f))
    {
        Motor_StopAll();
        s_context.error = MOTION_MANAGER_ERROR_BRAKE;
        return;
    }

    s_context.brakeElapsedSeconds += dt;
    brakeStartSeconds = MOTION_MANAGER_BRAKE_RELEASE_SECONDS;
    brakeEndSeconds = brakeStartSeconds + MOTION_MANAGER_BRAKE_HOLD_SECONDS;

    if (s_context.brakeElapsedSeconds < brakeStartSeconds)
    {
        /* 直线速度已平滑降至零；这里继续释放 PWM，避免突变制动。 */
        Motor_StopAll();
        return;
    }

    if (s_context.brakeElapsedSeconds < brakeEndSeconds)
    {
        Motor_Brake();
        return;
    }

    /* 制动脉冲结束必须释放方向脚，防止电机持续发热。 */
    Motor_StopAll();
    s_context.brakeFinished = 1U;

难点函数拆解

难点一：为什么每个行为都要有一个运行时结构体

输入：Start 写入一次性任务参数，Update 每拍写入进度，查询函数读取状态。

过程：结构体把同一任务的变量放在一起。Straight 的基线里程、总距离、速度曲线、目标航向和完成标志共同描述一段直线；Line 的上一拍轮速和 lostTicks 共同描述丢线过程；Nav 的目标角、斜坡转速和 settleCount 共同描述转向过程。

输出：函数即使返回了，下一拍仍能从静态 s_context 继续工作。

边界条件：启动新任务前必须 Reset 对应运行字段，否则上一任务的丢线计数、PID 状态或稳定计数会污染新任务；但 configured 应保留。

设计原因：比散落十几个静态变量更容易检查“哪些字段属于同一次动作”，也让状态转换和调试显示更清楚。

难点二：Straight 如何同时处理加速、定距、减速和终点速度

输入：带符号距离、巡航速度、终点速度、左右里程和 dt。

过程：启动时记录左右里程基线；每拍用两轮平均位移得到任务方向进度。前 5/6 距离以固定加速度靠近巡航速度；跨过减速点时，用当时实际规划速度和到 total-tolerance 的剩余距离反解恒定减速度；之后再用 v=sqrt(v_end²+2as) 产生距离相关目标速度。

例：总距离 600 mm、容差 5 mm，曲线终点是 595 mm，减速点是 500 mm。若越过减速点时实际速度 300 mm/s、终点速度 0，则剩余 95 mm，所需减速度约为 300²/(2×95)=473.7 mm/s²。以后越接近终点，平方根目标越低。

输出：同向的左右基础轮速，以及前进为正、后退为负的剩余距离。

边界条件：很短的任务可能来不及达到巡航速度，所以减速度必须基于“进入减速段时实际 profileSpeed”，不能盲用 cruiseSpeed。距离不超过容差且终点速度为 0 时可立即完成。

设计原因：时间斜坡负责平滑，距离曲线负责准确在终点速度收尾；两者分工避免“只按时间减速导致距离变化时到点速度不对”。

难点三：航向 PD 为什么作为 PWM trim，而不是改目标轮速

输入：启动时锁定的 targetHeadingDeg、当前 Heading_GetYaw() 和 dt。

过程：PD 计算航向误差对应的修正 PWM；基础左右目标轮速保持相同，而 trimL=-correction、trimR=+correction 形成相反修正。

输出：MotionWheel 先按左右目标速度做各自反馈，再叠加航向 trim，兼顾“平均前进速度”和“车头方向”。

边界条件：规划速度接近 0 时关闭航向修正，防止静止时因陀螺仪微漂而让轮子互相对转。trim 上限不能超过 MotionWheel 最终 PWM 上限。

设计原因：速度层专注每个轮子达到目标速度；车体航向是更高一层的差速修正。分层后调参含义更明确。

难点四：灰度加权巡线和丢线确认

输入：五位灰度位图与巡航速度。

过程：bit0/1 分别贡献 -6/-3，bit2 为 0，bit3/4 贡献 +3/+6；同侧叠加后钳在 ±6。归一化权重乘 20% 巡航速度得到左右一增一减的轮速差。全白时先保持上一拍命令并计数，连续 50 次 Update 才返回结束。

输出：左右不同的目标轮速；确认丢线时输出 FINISHED，而不是 ERROR。

边界条件：lostTicks 每次 Update 只加 1，没有按 dt 或 App 的 elapsedTicks 增加。正常 100 Hz 时 50 次约 0.5 s；若主循环一次跨过多拍，真实确认时间会变长。刚 Start 就全白时“上一拍速度”是 0，因此先静止等待确认。

设计原因：保持上一拍能穿过短暂反光、接缝或采样毛刺；把确认丢线定义为正常结束，Mission 可自然转入下一个状态。

难点五：绝对 TurnTo、相对 TurnBy 与“不会自动走最短路”

输入：Heading 的连续累计角、绝对目标或相对增量、允许轮速。

过程：TurnTo 直接保存目标；TurnBy 先计算 当前角 + 增量 再调用 TurnTo。每拍误差始终是 target-current，没有把它折回 -180°～+180°。

输出：误差为正时一种旋转方向，误差为负时相反；左右轮大小相等、符号相反。

边界条件：当前 370°、TurnTo(90°) 得到 -280°，而不是 +80°。想转最短路径应由调用者算出最短相对增量后调用 TurnBy。TurnBy 可明确要求 ±720° 等多圈动作。

设计原因：连续坐标没有跨 ±180° 的跳变，能准确表达多圈和 Mission 逐次累加的绝对目标。

难点六：到角后为什么还要“轮速归零 + 连续三拍”

输入：角误差、当前斜坡转速和稳定计数。

过程：角误差进 ±2° 后目标速度先变 0；斜坡用 600 mm/s² 减速。只有 profileTurnSpeed 也接近 0 才停止轮速并计一次稳定；连续 3 次达到才 COMPLETED。误差离开容差立即清计数。

输出：Nav finished 只在车已经基本停住且角度稳定约 30 ms 后出现。

边界条件：单拍冲进容差但仍高速旋转不会完成；惯性冲出容差会重新转向并从 0 开始稳定计数。

设计原因：仅判断位置误差会把“高速经过目标角”误判为完成；速度和时间条件能显著降低过冲误报。

难点七：MotionManager 的模式互斥、结果映射和刹车阶段

输入：Mission 的动作请求、当前 mode、子模块启动/运行结果和 dt。

过程：每个 Start 都先调用 MotionManager_Stop() 清掉旧模式，再启动新行为；成功才设置新 mode。Update 的 switch 每拍只调用一个行为。子模块不同的结果枚举先映射为统一 Manager 结果，运行错误再映射为 STRAIGHT/LINE/TURN 类别。Brake 则独占 mode，按“释放→主动刹车→释放”推进。

输出：Mission 只需面对一套 Start、Busy、Finished、Error 接口。

边界条件：Manager 的 Start 不是“忙就拒绝”，而是主动抢占：新命令会停止旧动作。切换失败时旧动作已经被停，新 mode 回 IDLE 并带错误。刹车 dt 无效时 error 被置位但 brakeFinished 仍为 0，Mission 必须先检查 error。

设计原因：唯一所有权防止多个行为同时写 MotionWheel；统一结果让 Mission 不必知道每个子模块的枚举细节。

难点八：Straight 的 finished 与 busy 为什么能同时为真

输入：终点速度是否为 0，以及 Straight 当前状态。

过程：零终点速度完成后进入 COMPLETED：finished=1、busy=0。非零终点速度到达分段终点后进入 CONTINUING：finished=1，告诉 Mission 可以转移；busy=1，因为本模块仍每拍维持终点速度。

输出：Mission 可把 CONTINUING 当作“这一段完成”，下一状态启动另一个动作接管。

边界条件：下一动作通过 Manager 启动时会先 Stop CONTINUING，轮速控制器与斜坡也会复位，因此不能假设跨动作绝对无缝继承内部 PID 状态。

设计原因：“任务边界是否到达”和“当前是否仍占用执行器”是两个独立问题，用两个查询比强行互斥更能表达真实状态。

放回总体调用链

下面出现的 onEnter、onUpdate 和动作状态适配属于 Mission 回调接口；本章只关注它们怎样启动运动，回调函数指针、转换规则和状态生命周期会在模块 08 系统解释。

Mission 状态 onEnter → MotionManager_StartForward/Backward → MotionStraight_Start → 每拍根据 Odometry/Heading 生成基础轮速与航向 trim → MotionWheel_Update → Motor

Mission 状态 onEnter → MotionManager_StartLine → MotionLine_Update 读取 Graydetect → 加权左右轮速 → MotionWheel_Update → Motor

Mission 状态 onEnter → MotionManager_TurnTo/TurnBy → Nav_Update 读取 Heading → 左右反向轮速 → MotionWheel_Update → Motor

定距零速完成 → Mission 下一个状态 → MotionManager_StartBrake → PWM 释放 → Motor_Brake 短脉冲 → Motor_StopAll

App.c 第 96～97 行先更新 Heading 和 Odometry，第 120 行才调用 MotionManager_Update(dt)，所以行为使用的是本拍新鲜状态。Mission 的 onUpdate 不应再次更新控制器，只需调用 Mission_GetMotionActionStatus()：Mission.c 第 286～298 行按“error 优先、finished 其次、busy 再次、其余视为完成”翻译动作状态。

三个行为共享 MotionWheel 的直接原因是它们最终都需要同一套左右轮速度 PID、前馈、PWM 限幅和 Odometry 反馈。如果各写一套，轮胎、电池或电机变化时要调三份参数，而且切换时可能残留三个不同的积分状态。共享的代价是必须严格互斥：Manager 的 switch 保证一拍只有一个生产者，所有启动入口也先 Stop 旧模式。

常见误解与调试提示

误解：请求速度就是电机立刻达到的速度

Straight 与 Nav 都有 profileSpeed。请求只是上限或远距离目标，每拍最多按加速度乘 dt 变化；MotionWheel 还要根据编码器反馈换算 PWM。

误解：减速参数就是固定的 300 mm/s²

Straight 的 300 是起步加速度。减速时在固定位置根据实际速度、终点速度和剩余距离动态反解 effectiveDeceleration，可能大于或小于 300。

误解：剩余距离为负就是计算错误

后退任务公开的 remainingDistance 带方向，因此未完成时为负；内部规划用的是正的距离大小。

误解：灰度中心 bit2 没参与巡线

它参与“是否看见线”和丢线判断，只是横向权重为 0，因此不会造成左右差速。中心单独触发时两轮都是巡航速度。

误解：NAV_MIN_TURN_SPEED 总会强制为 40 mm/s

代码把最小速度再钳到调用者请求速度。若请求 30，最低与最高都只能是 30；参数名表示正常请求范围内的低速下限。

误解：Manager 会拒绝正在忙时的新动作

Manager 没有公开 BUSY 结果。所有 Start 都先 Stop 当前模式再尝试新动作，所以新命令是抢占式切换。若新参数非法，旧动作也已经停止。

调试顺序

先单独验证 MotionWheel 的左右方向与速度反馈；再用低速 Nav 检查旋转 sign；再用短距离 Straight 看里程方向和剩余距离；最后才调航向 PD、巡线权重与刹车时间。上层异常经常源于共享轮速层，而不是行为公式本身。

修改参数会牵动哪里

  • 提高 Straight 加速度会缩短起步时间，但也提高打滑、过流和航向 PD 瞬态负担；短距离进入减速点时的实际速度也随之变化。
  • 改变距离容差同时改变“何时 targetReached”和速度曲线终点，Mission 的定距精度与零速时刻都会变化。
  • 改变航向 KP/KD 或 correction sign 会影响前进与后退直线；必须两方向都验证，且修正 PWM 上限不能超过 MotionWheel 上限。
  • 改变巡线权重或最大调整比例会改变最坏情况下左右速度差；比例过大可能让一侧速度过低、另一侧超过适合路面的速度。
  • 改变丢线确认拍数会影响 Mission 在黑线末端何时转移；当前实现按 Update 次数而不按真实 elapsedTicks。
  • 改变 Nav 低速、减速角、加减速度与容差会共同影响过冲和完成时间，不能只单独看一个值。
  • 改变 Manager 刹车保持时间会影响滑行距离、电机发热和顿挫；该模式绕过 MotionWheel，不能用轮速 PID 参数补救。
  • 任何行为切换都会复位或停止共享 MotionWheel；修改切换策略时必须重新审计唯一所有权，避免两个模块同拍写命令。

学完自测

1. Straight 为什么同时保存 totalDistance、remainingDistance 和 profileEndDistance？

total 是规划用的正总长度；remaining 是对外带方向的剩余量；profileEnd 是 total-tolerance，表示速度曲线应在进入允许误差区时已经达到终点速度。

2. 巡线位图只有最左 bit0 为 1、巡航速度 500 mm/s 时，左右目标速度是多少？

权重 -6，调整量为 500×0.2×(-6/6)=-100；左轮 500-100=400，右轮 500-(-100)=600 mm/s。

3. 当前连续航向 370°，Nav_StartTo(90°) 的初始误差是多少？怎样请求最短的 +80°？

误差是 90-370=-280°。若要明确走 +80°，调用 Nav_StartBy(80°)，而不是依赖 To 自动归一化。

4. Straight 进入 CONTINUING 时 IsBusy 和 IsFinished 分别是什么？

两者都是 1：本段距离已完成，所以 finished；但仍持续输出非零终点速度，所以 busy。

5. 为什么 MotionManager 启动新动作前必须先 Stop 旧模式？

Straight、Line、Nav 共享同一个 MotionWheel 和双电机。先 Stop 可清旧模块状态并确保同一时刻只有一个命令生产者；代价是新动作启动失败时旧动作也不会恢复。
