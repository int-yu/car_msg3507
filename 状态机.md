# Accomplish 状态机编写指南

本文用于指导新题目流程的编写。具体题目代码统一放在工程根目录 `Accomplish/`，通用调度器仍放在 `Application/Mission/`。

## 1. 各层职责

| 层级 | 目录或模块 | 职责 |
|---|---|---|
| 系统更新 | `Application/Core/App` | 每拍更新 MPU6050、里程、按键、蓝牙、MotionManager、蜂鸣器和 OLED |
| 通用状态机 | `Application/Mission` | 校验并执行静态状态图，不保存任何具体题目流程 |
| 题目实现 | `Accomplish/<题号>.c/.h` | 保存该题的参数、状态、条件、转换和动作回调 |
| 运动调度 | `Application/Control/MotionManager` | 保证任意时刻只运行直线、巡线或转向中的一种 |
| 运动算法 | `MotionStraight`、`MotionLine`、`Nav` | 实现具体运动控制 |

不要在 `main.c` 中直接编写比赛流程，也不要把 25E、25F 等题目的参数放进通用 Mission。

## 2. 运行时流程

主程序固定使用以下结构：

```c
int main(void)
{
    App_UpdateContext_t updateContext;

    App_Init();
    Mission_Init(Accomplish25H_GetMissionGraph());
    Interrupt_Enable();

    for (;;)
    {
        if (App_Update(&updateContext) != 0U)
        {
            Mission_Update(&updateContext);
        }
    }
}
```

每个有效系统节拍依次发生：

1. `App_Update()` 取得真实累计节拍并计算 `dt`。
2. App 更新航向、里程、按键边沿和蓝牙单次事件。
3. App 只更新 MotionManager 当前选中的运动模式。
4. `Mission_Update()` 调用当前题目状态的 `onUpdate(dt)`。
5. 动作仍在运行时，只检查打断转换。
6. 动作完成后，只检查正常转换。
7. 第一条满足条件的转换生效；每拍最多转换一次。
8. 转换时依次执行旧状态 `onExit()`、`MotionManager_Stop()` 和新状态 `onEnter()`。

`C0` 是通用全局停车信号，不受 `interruptible` 限制。它会停止车辆并回到题目状态图的 `startState`。

## 3. Accomplish 文件约定

每道题使用两个文件：

```text
Accomplish/
├── 25E.h       参数和唯一公共接口
├── 25E.c       状态编号、回调、条件、转换表和状态图
├── 25H.h       当前题目的全部可调参数和公共接口
├── 25H.c       当前题目的状态图实现
├── 其他题.h
└── 其他题.c
```

文件名可以使用 `25H.c`，但 C 函数和类型名称不能以数字开头，因此使用 `Accomplish25H_` 前缀。

题目 `.c` 内的状态编号、回调、转换表、状态表和运行计数全部使用 `static`。跨模块只公开一个取得状态图的函数。

## 4. 编写新题目的顺序

### 4.1 先画状态图

先把流程拆成有限状态，不要直接开始写回调。例如：

```text
WAITING --KEY1 按下--> STRAIGHT
STRAIGHT --检测到线--> LINE
STRAIGHT --距离走完--> WAITING
LINE --确认丢线--> TURN
TURN --转向完成--> STRAIGHT
```

每个状态只负责一个明确动作。不要在同一个 `onUpdate()` 中同时启动直线、巡线和转向。

### 4.2 在题目头文件集中参数

示例 `Accomplish/Example.h`：

```c
#ifndef ACCOMPLISH_EXAMPLE_H
#define ACCOMPLISH_EXAMPLE_H

#include "Application/Mission/Mission.h"

/* 距离单位 mm，速度单位 mm/s，角度单位 °。 */
#define ACCOMPLISH_EXAMPLE_START_KEY_MASK     0x01U
#define ACCOMPLISH_EXAMPLE_DISTANCE_MM        1000U
#define ACCOMPLISH_EXAMPLE_SPEED_MMPS         200.0f
#define ACCOMPLISH_EXAMPLE_TURN_ANGLE_DEG     90.0f
#define ACCOMPLISH_EXAMPLE_TURN_SPEED_MMPS    60.0f

const Mission_GraphDefinition_t *
AccomplishExample_GetMissionGraph(void);

#endif
```

头文件顶部只放需要用户调整的题目参数。MotionWheel、MotionLine、Nav 自身的控制参数仍放在各自模块头文件，不要复制到题目文件。

### 4.3 定义状态编号

状态编号只属于当前题目：

```c
typedef enum
{
    ACCOMPLISH_EXAMPLE_STATE_WAITING = 0,
    ACCOMPLISH_EXAMPLE_STATE_FORWARD,
    ACCOMPLISH_EXAMPLE_STATE_TURN,
    ACCOMPLISH_EXAMPLE_STATE_ERROR,
    ACCOMPLISH_EXAMPLE_STATE_COUNT
} AccomplishExample_StateId_t;
```

必须同时存在合法的起始状态和错误状态。所有转换目标都必须小于 `STATE_COUNT`，Mission 初始化时会检查这些编号。

### 4.4 编写进入回调

`onEnter()` 只调用一次，用于启动本状态的动作：

```c
static Mission_CallbackResult_t AccomplishExample_ForwardEnter(void)
{
    return (MotionManager_StartForward(
                ACCOMPLISH_EXAMPLE_DISTANCE_MM,
                ACCOMPLISH_EXAMPLE_SPEED_MMPS,
                0.0f) == MOTION_MANAGER_RESULT_OK) ?
        MISSION_CALLBACK_OK : MISSION_CALLBACK_ERROR;
}

static Mission_CallbackResult_t AccomplishExample_TurnEnter(void)
{
    return (MotionManager_TurnBy(
                ACCOMPLISH_EXAMPLE_TURN_ANGLE_DEG,
                ACCOMPLISH_EXAMPLE_TURN_SPEED_MMPS) ==
            MOTION_MANAGER_RESULT_OK) ?
        MISSION_CALLBACK_OK : MISSION_CALLBACK_ERROR;
}
```

启动失败必须返回 `MISSION_CALLBACK_ERROR`，Mission 会停车并进入题目定义的错误状态。

不要在 `onEnter()` 中写阻塞循环，也不要自行反复调用运动模块的 `Update()`。

### 4.5 编写更新回调

等待状态需要保持运行，才能检查 KEY1 按下沿等打断事件：

```c
static Mission_ActionStatus_t AccomplishExample_WaitingUpdate(float dt)
{
    (void)dt;
    return MISSION_ACTION_RUNNING;
}
```

普通运动状态统一查询 MotionManager：

```c
static Mission_ActionStatus_t AccomplishExample_MotionUpdate(float dt)
{
    (void)dt;
    return Mission_GetMotionActionStatus();
}
```

返回值含义：

| 返回值 | 含义 |
|---|---|
| `MISSION_ACTION_RUNNING` | 动作仍在运行，本拍只检查打断转换 |
| `MISSION_ACTION_FINISHED` | 动作已完成，本拍只检查正常转换 |
| `MISSION_ACTION_ERROR` | 动作失败，立即停车并进入错误状态 |

空 `onUpdate` 会被 Mission 当作“立即完成”。需要持续等待蓝牙或按键的状态必须提供返回 `RUNNING` 的更新回调。

### 4.6 编写条件回调

条件只回答“本条转换现在是否成立”，返回 1 表示成立：

```c
static uint8_t AccomplishExample_ConditionStart(
    const Mission_Runtime_t *runtime,
    const App_UpdateContext_t *updateContext)
{
    (void)runtime;
    if (updateContext == NULL)
    {
        return 0U;
    }

    return ((updateContext->pressedEdges &
             ACCOMPLISH_EXAMPLE_START_KEY_MASK) != 0U) ? 1U : 0U;
}
```

可使用的上下文数据包括：

```c
updateContext->elapsedTicks;
updateContext->dt;
updateContext->pressedKeys;
updateContext->pressedEdges;
updateContext->hasBluetoothSignal;
updateContext->bluetoothSignal;
```

蓝牙 `C1~C255` 是单拍事件。当前状态没有接受时会被丢弃，不会在以后补触发。

传感器条件容易抖动时，应在题目 `.c` 中使用静态计数确认，并在进入相关状态时清零。计数应累加 `elapsedTicks`，不能假定每次主循环固定经过 1 拍。

### 4.7 编写转换表

以下示例假设已经实现题目自己的 `AccomplishExample_ConditionDetected()`：

```c
static const Mission_Transition_t s_waitingTransitions[] = {
    {
        AccomplishExample_ConditionStart,
        ACCOMPLISH_EXAMPLE_STATE_FORWARD,
        MISSION_TRANSITION_INTERRUPT,
    },
};

static const Mission_Transition_t s_forwardTransitions[] = {
    {
        AccomplishExample_ConditionDetected,
        ACCOMPLISH_EXAMPLE_STATE_TURN,
        MISSION_TRANSITION_INTERRUPT,
    },
    {
        NULL,
        ACCOMPLISH_EXAMPLE_STATE_WAITING,
        MISSION_TRANSITION_NORMAL,
    },
};

static const Mission_Transition_t s_turnTransitions[] = {
    {
        NULL,
        ACCOMPLISH_EXAMPLE_STATE_WAITING,
        MISSION_TRANSITION_NORMAL,
    },
};
```

转换数组的声明顺序就是优先级，第一条满足条件的转换生效。

- `MISSION_TRANSITION_INTERRUPT`：只在动作返回 `RUNNING` 且当前状态允许打断时检查。
- `MISSION_TRANSITION_NORMAL`：只在动作返回 `FINISHED` 后检查。
- 条件为 `NULL`：该类型转换无条件成立。

如果同一传感器条件在动作完成的同一拍仍要生效，可以分别声明一条打断转换和一条正常转换，再把无条件兜底正常转换放在后面。

### 4.8 登记状态表

```c
static const Mission_StateDefinition_t s_stateDefinitions[
    ACCOMPLISH_EXAMPLE_STATE_COUNT] = {
    [ACCOMPLISH_EXAMPLE_STATE_WAITING] = {
        .onEnter = NULL,
        .onUpdate = AccomplishExample_WaitingUpdate,
        .onExit = NULL,
        .transitions = s_waitingTransitions,
        .transitionCount = 1U,
        .interruptible = 1U,
    },
    [ACCOMPLISH_EXAMPLE_STATE_FORWARD] = {
        .onEnter = AccomplishExample_ForwardEnter,
        .onUpdate = AccomplishExample_MotionUpdate,
        .onExit = NULL,
        .transitions = s_forwardTransitions,
        .transitionCount = 2U,
        .interruptible = 1U,
    },
    [ACCOMPLISH_EXAMPLE_STATE_TURN] = {
        .onEnter = AccomplishExample_TurnEnter,
        .onUpdate = AccomplishExample_MotionUpdate,
        .onExit = NULL,
        .transitions = s_turnTransitions,
        .transitionCount = 1U,
        .interruptible = 0U,
    },
    [ACCOMPLISH_EXAMPLE_STATE_ERROR] = {
        .onEnter = NULL,
        .onUpdate = NULL,
        .onExit = NULL,
        .transitions = NULL,
        .transitionCount = 0U,
        .interruptible = 0U,
    },
};
```

`interruptible=0` 只屏蔽普通打断转换，不会屏蔽 `C0` 全局停车。

通常不需要在 `onExit()` 中调用 `MotionManager_Stop()`，因为 Mission 转换时会统一停止当前运动。`onExit()` 只用于清理题目自身的计数、标志或其他非运动状态。

### 4.9 创建状态图描述

```c
static const Mission_GraphDefinition_t s_missionGraph = {
    .states = s_stateDefinitions,
    .stateCount = ACCOMPLISH_EXAMPLE_STATE_COUNT,
    .startState = ACCOMPLISH_EXAMPLE_STATE_WAITING,
    .errorState = ACCOMPLISH_EXAMPLE_STATE_ERROR,
};

const Mission_GraphDefinition_t *
AccomplishExample_GetMissionGraph(void)
{
    return &s_missionGraph;
}
```

状态表和图描述必须具有静态存储期，不能在函数栈上临时创建后返回。

### 4.10 在主程序选择题目

```c
#include "Accomplish/Example.h"

Mission_Init(AccomplishExample_GetMissionGraph());
```

一次只加载一个题目状态图。切换题目时只修改 Accomplish 头文件和传给 `Mission_Init()` 的图函数，不修改 App、MotionManager 或 Mission 主循环。

## 5. 正常完成、打断和错误的区别

| 情况 | 建议处理 |
|---|---|
| 直线途中检测到标志线 | 当前直线仍在运行，使用打断转换 |
| 直线走完目标距离 | MotionStraight 正常完成，使用正常转换 |
| “巡线直到丢线”确认丢线 | 这是任务预期结束条件，MotionLine 返回完成，使用正常转换 |
| MPU6050 掉线、更新周期非法、轮速控制失败 | 属于系统错误，进入错误状态 |
| 用户发送 `C0` | 全局停车并回到起始状态 |

不要把预期的流程条件误写成控制错误，否则 Mission 会先进入错误状态，后续题目转换不会执行。

## 6. 当前 25H 示例

当前 `main.c` 加载 `Accomplish/25H.c/.h`：

```text
WAITING
  └─ KEY1 按下沿 → LINE
LINE
  ├─ bit0 与 bit1 同时为 1 → FORWARD
  └─ 连续全白 50 拍 → WAITING
FORWARD
  └─ 直行 150 mm 并减速至零 → TURN
TURN
  └─ 到达本轮绝对左转目标 → LINE
```

25H 首次进入 LINE 时保存 MPU6050 连续航向作为 0°基准。每次进入 TURN 前把绝对目标减少 90°，然后调用 `MotionManager_TurnTo()`；目标依次为 `startYaw-90°`、`startYaw-180°`、`startYaw-270°`……。这里虽然每轮变化 90°，但传给 Nav 的始终是连续绝对角，不使用 `TurnBy()`。

左侧标志条件使用灰度掩码 `0x03U`，即 bit0 和 bit1 必须在同一拍同时为 1。该条件是运行中打断巡线，因此转换类型必须为 `MISSION_TRANSITION_INTERRUPT`，且 LINE 状态必须允许打断。FORWARD 和 TURN 都等待动作完成，使用正常转换。

所有 25H 用户参数都位于 `Accomplish/25H.h` 开头。若实车左转方向与角度符号约定相反，只修改 `ACCOMPLISH_25H_TURN_STEP_DEG` 的正负号，不要同时修改 Nav 内部方向参数。

## 7. 新题目接入检查表

1. 在 `Accomplish/` 新建题号 `.c/.h`。
2. 所有用户参数放在题目 `.h` 开头并使用唯一前缀。
3. 先画状态图，再写状态编号。
4. 每个运动状态只在 `onEnter()` 启动一次动作。
5. 运动状态的 `onUpdate()` 返回 MotionManager 状态。
6. 区分打断转换、正常转换和真正错误。
7. 为传感器抖动设置确认节拍，并使用真实 `elapsedTicks`。
8. 检查转换声明顺序是否符合优先级。
9. 检查 `startState`、`errorState` 和全部目标状态是否合法。
10. 在 `main.c` 选择新的状态图。
11. 同步修改 README 的时钟表、Pin 表、程序流程、文件职责、接口和参数。
12. 在 CCS 中刷新工程并执行 Clean/Rebuild，确保新建的 `.c` 已加入构建。
13. 先悬空测试电机方向和转向方向，再落地低速测试。

## 8. 常见错误

- 在等待状态把 `onUpdate` 留空，导致状态立即完成。
- 把完成后的转换写成打断转换，导致动作完成后不跳转。
- 把运行中的传感器触发写成正常转换，导致动作运行时条件从不检查。
- 在条件回调中启动电机；运动只能在目标状态的 `onEnter()` 中启动。
- 在多个状态中直接调用底层 `MotionStraight_Update()`、`MotionLine_Update()` 或 `Nav_Update()`；这些更新只能由 MotionManager 调度。
- 新建题目文件后没有刷新 CCS，导致链接时找不到 `Accomplish...GetMissionGraph`。
- 只修改代码，没有同步 README 中的硬件、流程和参数说明。
