# 串口调试网页与 K230 拍照 设计

日期：2026-07-18

## 1. 背景与目标

当前调试小车只能看 OLED 八行数据，信息量小、无法回放、无法画曲线。视觉题需要在真实车位、真实光照下采集素材图，但 K230 上没有按需拍照保存的入口。

本设计解决两件事：

1. 电脑网页通过无线串口控制小车运动（定距直行、定角转向、舵机、急停），并触发 K230 拍照。
2. 调试阶段把 MPU6050 航向、五路灰度、编码器里程与速度、按键、运动模式、K230 上报数据以 CSV 流式发到网页，供实时查看、画图和导出。

照片不回传电脑，直接存 K230 的 TF 卡，事后拔卡拷贝。这样彻底避开串口带宽问题。

## 2. 链路与硬件

只有一条对电脑的链路：无线 DAPLink 串口。

```
浏览器网页 (Web Serial API)
    │  UART1 / PB6-PB7 / 115200 8N1
    │  上行：CSV 日志行    下行：ASCII 命令
    ▼
MSPM0G3507
    │  UART2 / PA21-PA22 / 115200 8N1
    │  AA55 二进制帧（现有 K230Link 协议）
    ▼
K230
    └─ 存图到 TF 卡 /sdcard/pic/
```

网页不知道 K230 存在。拍照命令由 MSPM0 转成二进制帧转发，K230 回 ACK，MSPM0 再翻译成 ASCII 回给网页。K230 换程序不影响网页。

### 2.1 波特率变更

UART1 从 9600 改为 115200。同工程的 `D:\real\ti_25_e` 已在同一硬件（无线 DAPLink 接 PB6/PB7）上验证过 115200 可用，见其 `main.syscfg:194` 与 `README.md` 的 Wireless DAPLink serial 一节。

变更点只有 `main.syscfg` 的 `UART1.targetBaudRate`。`Hardware/Comms/Serial.c` 不需要改。

## 3. 控制权规则

`MotionManager` 只规定一件事：任意时刻只有一个运动模块（STRAIGHT / LINE / TURN / BRAKE）占用双轮。它不关心命令来源。

`App_Update()` 每拍无条件调用 `MotionManager_Update(dt)`（`Application/Core/App.c:118`），因此只要有人调过 `MotionManager_Start*`，运动就会执行，与 Mission 是否参与无关。

由此确定三条规则：

1. **网页运动命令在 `MotionManager_IsBusy()` 为真时一律返回 `ERR BUSY`。** 与现有 `L/R/U` 的规则一致。
2. **按键优先级高于网页。** 按键触发 Mission 状态转换时，Mission 会先执行 `MotionManager_Stop()` 再调用目标状态的 `onEnter`，天然把网页发起的运动顶掉。不需要额外代码。
3. **`C0` 全局停车保持最高优先级**，任何时候都能停车。

不新增 Accomplish 状态图，不修改 `main.c`。调试时沿用当前已加载的 `Accomplish/Test.c`：WAITING 状态不驱动车轮，KEY2 启动定距直行加短暂刹车。网页命令在 WAITING 期间独占运动层。

### 4.2 命令集

现有 `C/L/R/U/O/D` 全部保留，语义不变。新增：

| 命令 | 作用 | 范围 | 忙时行为 |
|---|---|---|---|
| `F<mm>` | 前进定距，终点速度 0 | 1~10000 | `ERR BUSY` |
| `B<mm>` | 后退定距，终点速度 0 | 1~10000 | `ERR BUSY` |
| `T<deg>` | 相对转角，带符号 | -3600~3600 | `ERR BUSY` |
| `A<deg>` | 绝对连续航向角 | -3600~3600 | `ERR BUSY` |
| `Z<n>` | `n=1` 时把当前朝向定为 0° 基准 | 1 | `ERR BUSY` |
| `V<mmps>` | 设置调试巡航速度 | 20~800，默认 200 | 随时可改 |
| `P<n>` | 请求 K230 连拍 n 张 | 1~20 | 随时 |
| `G<hz>` | 日志频率，0 为关闭 | 0~100，默认 20 | 随时 |
| `M<mask>` | 日志字段掩码 | 0~63，默认 63 | 随时 |

字母与现有命令无冲突。`D` 已被横向舵机占用，因此绝对航向角用 `A`。

解析器约束：`BluetoothDebug_ExecuteCommand()` 要求 `hasDigits` 非零，否则返回 `ERR FORMAT`。因此拍照写作 `P1` 而非 `P`，顺带天然支持 `P5` 连拍。新命令字母必须加入 `BluetoothDebug_IsCommand()`。

`T` 与 `A` 的参考系区别（`Nav` 不做 ±180° 最短路径优化，见 README 3.7）：

```
当前航向 370°
  T90  →  相对再转 90°  →  终点 460°
  A90  →  转到绝对 90°  →  倒转 280° 回到 90°
```

`Z1` 调用 `Heading_SetYaw(0.0f)`。它必须检查 `MotionManager_IsBusy()`：README 3.4 指出 `MotionStraight` 运行期间重置航向会改掉本次直线的航向基准，导致跑偏。

`F/B/T/A` 的实现统一为：检查忙 → 检查范围 → 调用对应 `MotionManager_Start*` / `TurnTo` / `TurnBy`，速度取当前 `V` 值，终点速度固定 0 → 回 `OK` 或 `ERR`。

### 4.3 日志格式

CSV 文本行，字段随 `M` 掩码变化。掩码改变时立即重发一行表头，网页据表头解析后续数据行：

```
H,ms,yaw,gray,keys,LD,RD,LV,RV,mode,k230
D,12340,-91.25,1F,0,1203.4,1198.7,0.0,0.0,IDLE,1:+123:-45
D,12390,-91.30,1F,0,1203.4,1198.7,0.0,0.0,IDLE,0:0:0
```

字段掩码位定义：

| 位 | 值 | 字段 |
|---|---:|---|
| bit0 | 1 | `yaw` 连续累计航向角 |
| bit1 | 2 | `gray` 五路灰度位图（十六进制）与 `keys` 按键位图 |
| bit2 | 4 | `LD` / `RD` 左右轮累计路程 mm |
| bit3 | 8 | `LV` / `RV` 左右轮实测速度 mm/s |
| bit4 | 16 | `mode` 运动模式文本 |
| bit5 | 32 | `k230` 最近一次 TARGET，格式 `valid:x:y` |

带宽核算，115200 8N1 约 11520 字节/秒，全字段一行约 75 字节：

| 频率 | 占用 | 说明 |
|---:|---:|---|
| 20 Hz（默认） | 13% | 命令响应无感 |
| 50 Hz | 33% | 常规调参足够 |
| 100 Hz（上限） | 65% | 可跑，命令响应有抖动 |

`Telemetry` 只做组装与限频，数据全部通过现有公共接口读取（`Heading_GetYaw`、`Graydetect`、`Odometry_*`、`MotionManager_GetMode`、`K230Link_GetTarget`），不自己采集，不缓存历史。

### 4.4 拍照协议

复用现有 `AA55 | VER | TYPE | SEQ | LEN | PAYLOAD | CRC8` 帧，新增两个 TYPE：

```
0x20 CAPTURE      MSPM0 → K230   PAYLOAD: count:u8                （1 字节）
0x21 CAPTURE_ACK  K230 → MSPM0   PAYLOAD: ok:u8, index:u16_LE     （3 字节）
```

`index` 为本次保存的最后一张图片的编号。MSPM0 收到 ACK 后向网页回 `OK CAP <index>`；超过 1 秒未收到 ACK 回 `ERR CAP TIMEOUT`；`ok=0` 回 `ERR CAP SAVE`。

握手不变，仍为 READY / READY_ACK 双向握手。`K230Link_RequestCapture()` 在握手未完成时直接返回失败，网页收到 `ERR CAP NOLINK`。

### 5.3 存图 API 风险

README「板端使用精简版 OpenCV」一节列出的已验证函数中**没有 `imwrite`**。因此 `save()` 优先使用 CanMV image 对象自带的 `image.save(path)`（OpenMV 风格 API），而不是 `cv2.imwrite(frame)`。

代价是 `save()` 接收 `image` 对象而非 numpy `frame`。两条路径都实现并加显式报错，首次上板即可确定哪条可用，确定后删掉另一条并更新本节。

### 5.4 `uart_io.py` 回调注册

`_handle_frame()` 当前用 if/elif 直接处理 READY / READY_ACK / TARGET。新增一个处理器字典：

```python
def set_command_handler(self, message_type, callback):
    """注册非协议层消息的处理器。callback(payload) -> bytes 或 None。"""
```

`_handle_frame()` 在现有分支之后查表分发。返回值非 None 时作为 ACK payload 自动回帧。

这样 `uart_io.py` 仍然只负责协议编解码与握手，不引入文件系统和图像依赖，符合现有分层。

## 6. 网页

`car_msg3507/car_debug.html`，单文件，无外部依赖，Web Serial API（Chrome / Edge，需 HTTPS 或 file:// 本地打开）。

四个区域：

1. **连接区**：选择串口、115200 连接、连接状态、`C0` 急停大按钮。
2. **控制区**：距离 / 角度 / 速度输入框，前进 / 后退 / 左转 / 右转 / 归零按钮，两个舵机滑块，拍照按钮与张数。
3. **日志区**：实时数值表格、yaw 与左右轮速度折线图、CSV 导出、频率与字段掩码控制。
4. **原始终端**：收发原文，便于排查协议问题。

网页只依赖第 4.2 节的 ASCII 命令集与第 4.3 节的 CSV 格式，不含任何 K230 知识。

## 7. 验证顺序

1. 改波特率，只发 `V200`，确认回 `OK`，确认链路通。
2. `G0` 关日志，逐条测 `F/B/T/A/Z`，用 OLED 和卷尺核对。
3. `G20` 开日志，确认表头与数据行格式，确认网页能画曲线。
4. `M` 掩码逐位测试，确认表头随之变化。
5. 恢复 K230Link，确认 OLED 能显示握手成功。
6. K230 单独跑 `capture.py`，确认存图 API 可用，确认编号递增。
7. 联调 `P1`，确认 `OK CAP <index>` 与 TF 卡文件一致。
8. `G100` 压测，确认命令仍可响应。
