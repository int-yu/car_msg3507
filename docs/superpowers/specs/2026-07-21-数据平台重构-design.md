## 1. 平台定位（用户确认）

这个平台有**两个并列目的**，不只是调参：

| 目的 | 说明 | 对系统的要求 |
|---|---|---|
| **A. 遥控采数据** | 像自动驾驶那样遥控小车跑，收集数据供后续任务（视觉、决策、模型）使用 | 数据要**完整、无损、带准确时间戳**、能长时间录制、能批量导出 |
| **B. 调参** | 轮速 PI、直线航向 PD、转向、巡线的闭环整定 | 采样率要**高于控制环**、目标 vs 实测同图、指标自动化 |

两个目的对"数据质量"的要求是一致的，因此本设计以**数据链路**为核心。

## 2. 现状诊断（已查证的事实，非推测）

### 2.1 硬件与资源

| 项 | 实测值 | 来源 |
|---|---|---|
| SRAM 总量 | 32 KB（0x8000） | `Debug/ArcLineTest.map` MEMORY CONFIGURATION |
| SRAM 已用 | 3.5 KB（0xdc1） | 同上 |
| **SRAM 空闲** | **29 KB（0x723f）** | 同上 |
| FLASH 已用 | 42 KB / 128 KB | 同上 |
| UART1 波特率 | 115200 8N1 | `main.syscfg:194` |
| UART1 中断 | **仅 RX**，TX 中断与 DMA 均关闭 | `main.syscfg:191-193` |
| 控制环频率 | 100 Hz（10 ms） | `System/Tick.h` |

### 2.2 三个根本问题

**问题 1：采样率低于控制环 3.3 倍 —— 调参数据本身失真**

```
控制环   100 Hz  ← 实际动态
遥测采样  30 Hz  ← 当前能看到的
```

一次轮速阶跃上升沿约 100~200 ms，30 Hz 只能采到 3~6 个点，**超调峰值大概率落在采样点之间**。
基于这种数据算出的"超调 10%"可能实际是 18% 或 4%，调 P 增益等于蒙眼。

行业对标印证了这一点：
- [Betaflight Blackbox](https://betaflight.com/docs/wiki/guides/current/Black-Box-logging-and-usage)：PID tab 实时曲线只用于"确认电机有响应"，真正调参必须用 2 kHz 板载记录。
- [ODrive](https://docs.odriverobotics.com/v/latest/guides/tuning.html)：板载 oscilloscope 先写 buffer，再整体 dump。

**问题 2：阻塞发送 —— 遥测与控制互相伤害**

`Serial1_SendByte()` 用 `DL_UART_Main_transmitDataBlocking()`，
发送期间**主循环完全停止**，`Heading_Update` / `Odometry_Update` / `MotionManager_Update` 全部挂起。
所谓"20% 带宽上限"（`TELEMETRY_MAX_BLOCKING_PERCENT`）本质是**在用降采样掩盖阻塞缺陷**。
这也是目的 A（长时间采数据）的硬伤：采得越密，控制越不准。

**问题 3：ASCII CSV 协议效率低**

一个 float 占 10~11 字节（`,-999999.99`），二进制 float32 只要 4 字节。带宽利用率不足 40%。

### 2.3 用户直接提出的两个问题（均已定位）

**"调左右轮时应该只发相关数据"** —— 网页 `TUNING_FOCUS` 已按左/右切换曲线（`['TL','LV','PL']`），
但**固件掩码粒度是"双轮成对"**（`TELEMETRY_FIELD_SPEED` = LV+RV 绑定），
选"左轮"时右轮三列仍在传输，纯属浪费。

**"根据字段自动给最大频率"** —— 固件 `Telemetry_GetMaxRateHz()` 早就精确算出上限，
但只在**超限被拒时**通过 `ERR RANGE MAX=<n>` 吐出。
网页只能用 `SESSIONS` 表里硬编码的 `rate: 30` 猜，固件改一行常量就失配。

### 2.4 带宽实测账（node 计算，非估计）

| 场景 | 行长 | 上限 |
|---|---:|---:|
| 全字段 ASCII | 158 B | 14 Hz |
| 轮速试验子集 ASCII | 75 B | 30 Hz |
| 单侧理想 ASCII（TL,LV,PL） | 40 B | 57 Hz |
| **单侧二进制 float32** | **19 B** | **121 Hz** |
| **板载缓冲（20KB）** | 16 B/样本 | **100 Hz 无损，可存 12.5 秒** |

## 3. 目标架构：双通道

借鉴 ODrive / Betaflight 的成熟范式，把「监视」与「分析」分成两条路。

```
                   ┌─────────────────────────────────────┐
                   │            固件 100 Hz 主循环         │
                   └──────────────┬──────────────────────┘
                                  │
              ┌───────────────────┴────────────────────┐
              ▼                                        ▼
   ┌──────────────────────┐              ┌──────────────────────────┐
   │ 通道 A：实时流        │              │ 通道 B：板载高速捕获       │
   │ 10~20 Hz、ASCII CSV  │              │ 100 Hz 无损写 RAM 环形缓冲 │
   │ 用途：监视/状态/急停   │              │ 用途：调参分析 + 数据采集   │
   │ 非阻塞 TX（改造后）    │              │ 动作结束后一次性 dump      │
   └──────────────────────┘              └──────────────────────────┘
```

**为什么通道 B 是关键**：捕获期间**一个字节都不发串口**，主循环零阻塞，
数据以完整控制环频率落 RAM；车停下后才 dump，此时阻塞无害。
这同时解决问题 1 和问题 2，且对目的 A 和 B 都有效。

### T1 ⭐ 固件：非阻塞 TX 环形缓冲

**动机**：消除问题 2。这是所有后续任务的地基——不解决阻塞，
任何提高遥测频率的努力都会以牺牲控制精度为代价。

**改动**：
- `main.syscfg`：`UART1.enabledInterrupts = ["RX", "TX"]`（保留 RX，新增 TX）
- `Hardware/Comms/Serial.c/.h`：
  - 新增 TX 环形缓冲（建议 512 B，`SERIAL1_TX_BUFFER_SIZE`）
  - `Serial1_SendByte()` 改为写缓冲 + 使能 TX 中断，立即返回
  - TX 中断服务程序中逐字节喂 UART，缓冲空时关闭 TX 中断
  - 缓冲满时的策略：**丢弃新数据并计数**（不可阻塞等待，否则退化回原状）
  - 新增 `Serial1_GetTxDropCount()` 供诊断
- `Application/Debug/Telemetry.c`：`TELEMETRY_MAX_BLOCKING_PERCENT` 语义改为
  "TX 缓冲占用率上限"，安全频率上限可显著放宽

**验收**：
- 遥测开到 50 Hz 全字段时，`MotionManager` 的 dt 抖动不增大（用 `ms` 列差值验证）
- TX 丢弃计数在正常频率下恒为 0
- 现有 32 条测试全过

**风险**：TX 中断与主循环共享缓冲索引，需用 `volatile` + 单生产者单消费者无锁模式
（与现有 RX 缓冲同构，可直接照抄模式）。

### T2 ⭐ 固件：板载高速捕获（Capture Buffer）

**动机**：解决问题 1。这是本次重构的**核心价值**——
采样率从 30 Hz 提到 100 Hz，且对目的 A（采数据）同样是质变。

**新模块** `Application/Debug/Capture.c/.h`：

```c
#define CAPTURE_MAX_CHANNELS   4U
#define CAPTURE_BUFFER_BYTES   20480U   /* 20 KB，留 9 KB 余量 */

typedef enum { CAPTURE_STATE_IDLE, CAPTURE_STATE_ARMED,
               CAPTURE_STATE_RUNNING, CAPTURE_STATE_FULL } Capture_State_t;

void     Capture_Init(void);
uint8_t  Capture_Arm(uint16_t channelMask);   /* 选通道，进入待触发 */
void     Capture_Trigger(void);               /* 由运动命令自动调用 */
void     Capture_Update(void);                /* 每个 100Hz 拍调用，写一帧 */
void     Capture_Stop(void);
uint16_t Capture_GetSampleCount(void);
void     Capture_DumpNext(void);              /* 每拍吐一小段，避免长阻塞 */
Capture_State_t Capture_GetState(void);
```

**存储格式**（每样本 4 + 4×N 字节，N = 通道数）：
`uint32_t ms` + `float32 × N`

**通道定义**（复用现有遥测字段语义，但**单侧独立**）：

| 位 | 通道 | 来源 |
|---:|---|---|
| 0x01 | TL | `MotionWheel_GetTargetSpeedL()` |
| 0x02 | LV | `Odometry_GetSpeedL()` |
| 0x04 | PL | `MotionWheel_GetLeftCommandPWM()` |
| 0x08 | TR | `MotionWheel_GetTargetSpeedR()` |
| 0x10 | RV | `Odometry_GetSpeedR()` |
| 0x20 | PR | `MotionWheel_GetRightCommandPWM()` |
| 0x40 | yaw | `Heading_GetYaw()` |
| 0x80 | navE | `Nav_GetAngleErrorDeg()` |
| 0x100 | lerr | `MotionLine_GetLineError()` |

最多同时选 `CAPTURE_MAX_CHANNELS` 个（4 个 → 20 字节/样本 → 1024 样本 → **10.2 秒 @100Hz**）。

**接入点**：`App_Update()` 中在 `MotionManager_Update()` 之后调用 `Capture_Update()`
（确保采到的是本拍最终值）。

**验收**：
- `X` 捕获 8 秒 @100 Hz 不丢点（dump 出的 ms 列步长恒为 10）
- 捕获期间 UART1 无任何输出
- 捕获期间主循环 dt 与不捕获时无差异

### T3 ⭐ 固件：X / Q 命令

**动机**：T2 的控制接口 + 解决用户提的"自动频率"问题。

**新命令**（`Application/Comms/BluetoothDebug.c`）：

| 命令 | 作用 | 回应 |
|---|---|---|
| `X<mask>` | 按通道掩码 arm 捕获；下一个运动命令自动触发 | `OK X ARM=<mask> CAP=<最大样本数>` |
| `X0` | 停止捕获并 dump 全部数据 | `OK X DUMP=<n>` 后跟 `C,<ms>,<v1>,...` 数据行，末尾 `OK X END` |
| `Q` | 查询遥测能力与当前状态 | `OK Q MAX=<上限Hz> MASK=<掩码> RATE=<当前Hz> CAP=<捕获状态>` |

`Q` 是解决"自动给最大频率"的正解：网页**每次切换掩码后主动查询**，
拿到固件真实计算的上限，不再硬编码。

**dump 格式**：ASCII CSV（`C,` 前缀），复用网页现有解析器骨架，
每拍吐若干行避免长时间阻塞（`Capture_DumpNext()`）。

**验收**：`X7` → `F500` → 自动触发 → 动作完成 → `X0` → 收到完整数据；`Q` 回报与实际一致。

### T4 🔧 固件：遥测掩码细化到单侧

**动机**：直接解决用户提的第一个问题（"只发相关数据"）。

```c
#define TELEMETRY_FIELD_SPEED_L   0x400U   /* 只发 LV */
#define TELEMETRY_FIELD_SPEED_R   0x800U   /* 只发 RV */
#define TELEMETRY_FIELD_TARGET_L  0x1000U  /* 只发 TL */

#define TELEMETRY_FIELD_PWM_R     0x8000U  /* 只发 PR */

```

**验收**：`M0x1400`（TL+LV）行长约 36 B，`Q` 回报上限 ≥ 60 Hz。

### T6 🔧 网页：自适应频率

**动机**：解决用户提的第二个问题。

**改动**：
- 删除 `SESSIONS` 里硬编码的 `rate` 字段
- 每次应用掩码后自动发 `Q`，用回报的 `MAX` 设置频率（取 `min(MAX, 期望值)`）
- 频率输入框旁实时显示"当前掩码上限 <n> Hz"，超限时输入框变色而非等固件报错
- 掩码勾选变化时**即时**重算并显示预估上限（本地按同一公式算，与固件对账）

**验收**：勾选/取消字段时上限数字实时变化，与固件 `Q` 回报一致。

### T8 ⭐ 网页：数据采集模式（服务目的 A）

**动机**：用户的第一目的是"遥控采数据"，当前平台完全没有为此设计。

**新增**：
- **录制会话**：一键开始/停止，全程记录，显示时长、样本数、数据量

**验收**：能遥控跑 2 分钟、中途打 3 个标记、导出带标记列的完整 CSV。

```
                ├──> T2（捕获缓冲）──> T3（X/Q命令）──┬──> T5（网页捕获）──> T8（采集模式）
```

**最小可用集**：T2 + T3 + T5（跳过 T1，捕获期间本来就不发串口，阻塞问题不影响捕获质量）。
若额度有限，优先做这三个，能立刻把调参数据质量从 30 Hz 提到 100 Hz。

**T1 的价值**在于让通道 A（实时流/遥控采数据）也能高频且不伤控制，服务目的 A。

## 6. 明确不做

- 巡线改连续 PID（结构重构，另开分支）
- 参数写 Flash 持久化（用网页 localStorage 回写代替，已实现）
- 二进制遥测协议（捕获模式已解决采样率问题，ASCII 的可调试性更值钱）
- 串口控制无刷云台（固件中尚无该模块，用户已确认跳过）
- 舵机相关（用户要求忽略）

## 7. 已知约束

- MSPM0G3507 SRAM 32 KB，捕获缓冲取 20 KB，须为栈和后续功能留 9 KB
- 捕获通道数上限 4，超过会显著缩短可录时长
- dump 期间小车必须静止（阻塞发送，且此时无控制需求）
- 网页需 Chrome/Edge（Web Serial API）
